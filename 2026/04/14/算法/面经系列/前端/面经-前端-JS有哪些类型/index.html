面经-前端-JS 里有哪些类型

飞向月球的猫2026-04-142026-05-18

Q1

题目： JavaScript 里有哪些类型？

答：

从 ECMAScript 规范 的角度，数据分为两大类：

  1. 原始类型（Primitive，常叫「基础类型 / 简单类型」）
    共有 7 种：undefined、null、boolean、number、bigint、string、symbol。
    它们是不可变的值；赋值或传参时，传递的是值的拷贝（就语义而言）。

  2. 对象类型（Object，常和「引用类型 / 复杂类型」混用）
    规范里 Object 是一大类，普通对象、数组、函数、日期、正则等本质上都属于 Object 或由其派生。
    变量里通常存的是指向堆上对象的引用，多个变量可以指向同一对象。

需要纠偏的几点（面试常踩坑）：

  • NaN 不是单独一种类型，它是 number 类型里的一个特殊数值（typeof NaN === 'number'）。
  • null 属于原始类型，不要因为它和「对象引用」一起出现就把它归到对象里；历史上 typeof null === 'object' 是语言遗留问题，不能据此认为 null 是对象类型。
  • 现代 JS 别忘了 symbol（ES2015）和 bigint（ES2020），答全会显得基础更扎实。

栈 / 堆怎么说更稳妥：
教材里常说「简单类型在栈、对象在堆」是一种便于理解的模型。真实引擎（V8 等）会做优化：标量可能放栈或寄存器、小对象与逃逸分析都会影响分配位置。面试可以答：语义上是按值 vs 按引用；实现上对象主体一般在堆上，由引用访问。

追问 1：为什么 typeof null === 'object'？
早期实现里，null 被当成「空对象引用」的标记，在底层与对象指针表示相近；typeof 的判定表沿用了这一行为，成了改会破坏兼容性的历史包袱。规范里 null 仍是原始类型，与「是不是对象」的分类要以语言类型系统为准，不能只看 typeof。若需要更可靠的内置类型标签，可用 Object.prototype.toString.call(x)（如 [object Null]、[object Array]），或配合 Array.isArray 等 API。

追问 2：typeof 还有哪些容易混的？

  • typeof undefined → 'undefined'
  • typeof function () {} → 'function'（单独一类，便于区分可调用对象；本质仍是对象语义上的 callable）
  • typeof [] / typeof {} → 都是 'object'（数组要配合 Array.isArray）
  • typeof NaN → 'number'

追问 3：赋值、浅拷贝、深拷贝和类型有什么关系？

  • 原始类型：赋值 / 传参传的是值的拷贝，改一个变量不会影响另一个。
  • 对象：变量里是引用；a = b 只是两个变量指向同一对象，改对象属性双方都能看到。
  • 浅拷贝（如 {...obj}、Object.assign、slice）：最外层是新容器，但嵌套对象仍是引用共享，改深层仍可能互相影响。
  • 深拷贝：把每一层引用都换成新对象；实现上常用 递归遍历 + WeakMap 记已拷贝节点（处理循环引用）、或引擎提供的 structuredClone、或成熟库（如 lodash 的 cloneDeep）。
  • 不要用「JSON.stringify + JSON.parse」当通用深拷贝：它只能表达 JSON 规范里的那类纯数据，和「内存里的任意 JS 值」不是一回事。

为什么 JSON 序列化覆盖不了所有场景？（本质：JSON 是文本格式 + 类型子集，不是 JS 值的完整镜像。）

  1. 会丢的值：undefined、函数、Symbol（作键或作值）在 JSON.stringify 里会被忽略或变成 undefined 再被丢掉；单独 undefined 序列化结果是 undefined（连合法 JSON 文本都不是）。
  2. 类型被「压扁」再「误解」：Date 变成 ISO 字符串，parse 回来是普通字符串而不是 Date；NaN、Infinity 会变成 null。
  3. 不支持的结构：Map、Set、RegExp、ArrayBuffer、类实例上的原型链 / 自定义类身份等，JSON 里没有对应表示，序列化后要么丢失要么只剩「能转成普通对象的那部分可枚举属性」。
  4. 循环引用：对象 a 通过属性又指回自己时，JSON.stringify(a) 直接抛错。
  5. bigint：默认 stringify 会抛错，除非自己写 replacer 转成字符串再自己约定怎么 parse 回来。

**那怎么做序列化 / 深拷贝？**按需求选工具即可：

场景 做法
现代浏览器 / Node，可结构化克隆的数据（无函数、多数内置类型有支持） structuredClone(obj)，循环引用也能处理（会抛 DataCloneError 的项再换方案）。
只要 JSON 能表达的那类配置（纯对象、数组、字符串、数字、布尔、null） JSON.stringify，需要日期等可配合 replacer / reviver 做白名单与还原（仍要手写约定）。
任意复杂对象、要稳 递归深拷贝（处理 Map/Set/循环引用等分支）或 lodash.cloneDeep 等经过验证的实现。
跨端、与人可读 JSON / YAML 等；接受「只传 JSON 子集」的前提，并在文档里写清字段含义与可选类型。

面试里一句话收束：JSON 是交换格式，不是拷贝算法；深拷贝要按「能不能丢函数 / 原型 / 特殊对象」选型。
整体上把「类型 → 按值还是按引用 → 拷贝层次」串起来答，比只背类型名更有说服力。

追问 4：函数到底算哪种？
日常说法里常把函数和 Object、Array 一起叫「引用类型」。严格说，函数是可调用对象，有 [[Call]]；typeof 给 'function' 是为了用起来方便。答「引用 / 对象一侧」即可，若被追问再落到 callable 与 prototype 链上。
