ObjectWrap

nodejs最早期的版本代码量还没有这么多,动辄千万行,可以做一个全面的理解,对最精髓的部分做到把控

其实也就是说,这里的绑定方式是:在构造时把 this 塞进 JS 对象的 InternalField[0]（v8::External），并保存一个 Persistent <Object> 到 handle_。handle_.MakeWeak(…) 把持久句柄设为弱引用；当 JS 侧不再强引用该对象时，V8 会回调 MakeWeak，此时若未“附着”，即可删除 C++ 实例。

这里有双向绑定,将this指针绑定到v8的内部0字段;将v8永生句柄绑定到handle_指针.这正是Node.js实现JavaScript与C++互操作的核心机制，确保了两个层面对象之间能够相互访问和操作。

再来解释一下为什么这里叫弱引用的回调:

两套独立的引用计数系统

  1. JavaScript引用计数 ：
  • 由V8引擎管理
  • 当JavaScript代码中不再引用对象时，V8会触发 MakeWeak 回调
  • 此时设置 weak_ = true
  1. C++引用计数 ：
  • 通过 attach_count_ 手动管理
  • 通过 Attach() 增加，Detach() 减少
  • 只有当 weak_ = true 且 attach_count_ == 0 时才会真正删除对象

对象清理条件

对象只有在满足以下两个条件时才会被真正清理：

  1. weak_ == true (JavaScript层面已进入弱引用状态)
  2. attach_count_ == 0 (C++层面没有活动引用)

这种设计确保了：

  • 避免悬空指针：即使JavaScript对象被回收，如果C++层仍在使用就不会被删除
  • 防止内存泄漏：确保两个层面都不需要的对象能够被正确清理
  • 安全的双向访问：保证JavaScript和C++代码都能安全地访问共享对象

整体生命周期

关键点

  • JS ⇄ C++ 桥：InternalField[0] 放 External(this)；Unwrap() 取回。
  • GC 安全阀：MakeWeak + attach_count_，忙时不回收，闲时可回收。
  • 内存记账：InformV8ofAllocation(+size)，析构前注意 Adjust…(-size)（本项目在析构路径中调用）。
  • 使用手法：子类在构造时调用基类构造，暴露 JS 方法时先 Unwrap() 再操作原生对象。
