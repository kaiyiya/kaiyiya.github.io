一、net模块

1. 回顾HTTP请求

1）HTTP请求模式
  • 两种模式：分为普通模式（传统版本）和长连接模式，主要区别在于连接建立后的数据传输方式
  • 底层协议：HTTP协议建立在TCP/IP协议之上，三次握手和四次挥手由TCP/IP协议完成
2）普通模式

  • 通信流程：
    • 三次握手：客户端与服务器建立连接（”你在吗？-我在-你也在吗？-我也在”）
    • 请求响应：客户端发送请求 → 服务器返回响应
    • 四次挥手：完成通信后断开连接（实际应为四次挥手而非握手）
  • 特点：
    • 每次请求都需要完整的握手和挥手过程
    • 浏览器和操作系统自动处理底层连接，开发者只需关注请求和响应
    • 传统模式效率较低，特别是在需要多次请求时
3）长连接模式

  • 工作原理：
    • 通过Connection: keep-alive头保持连接不立即关闭
    • 客户端告知服务器”不要急着挂电话”，服务器响应确认保持连接
  • 优势：
    • 减少重复握手和挥手次数（如网页加载多个资源时）
    • 提高传输效率，降低延迟
  • 实现方式：
    • 客户端请求头包含Connection: keep-alive
    • 服务器响应头也包含Connection: keep-alive确认保持连接
  • 终止条件：
    • 服务器等待一段时间无新请求后主动关闭
    • 客户端确认无后续请求后主动关闭
    • 超时自动断开
2. net模块基础

1）模块功能
  • 本质：Node.js中的底层通信模块
  • 两种应用：
    • 进程间通信(IPC)：操作系统进程间的通信协议（前端较少使用）
    • 网络通信(TCP/IP)：基于TCP/IP协议的网络通信（重点学习内容）
2）与HTTP协议对比

  • 通信方式：
    • HTTP：严格遵循请求-响应模式，一次请求对应一次响应
    • TCP：建立连接后双方可自由互发数据，无严格次序限制
  • 灵活性：
    • HTTP：协议规定数据格式和交互方式
    • TCP：可传输任意格式数据，支持多次双向通信
  • 应用场景：
    • HTTP：适用于标准的Web请求响应场景
    • TCP：适用于实时通信、自定义协议等场景
3. 创建TCP客户端

1）基本创建方法
  • 核心方法：net.createConnection(options[, connectListener])
  • 返回值：返回socket对象用于后续通信
  • 参数说明：
    • options：包含端口、主机等连接配置
    • connectListener：连接成功后的回调函数
4. net模块能干什么
1）创建客户端

  • net.createConnection

    • 参数配置:
      • 第一个参数为配置对象，必须包含host和port两个关键属性
      • host表示要连接的主机地址（如”duyi.ke.qq.com”），注意不需要写协议头
      • port表示端口号（如8080），TCP/IP协议必须明确指定端口号
    • 连接监听:
      • 第二个参数是连接成功后的回调函数
      • 示例：net.createConnection({host:”duyi.ke.qq.com”, port:80}, ()=>{console.log(“连接成功”)})

  • socket

    • socket的概念

      • 本质:
        • 是操作系统中的特殊文件
        • 与网卡直接关联，负责网络端口的数据传输
      • 功能特点:
        • 建立TCP/IP连接后返回的对象
        • 客户端通过socket与远程主机通信

    • socket在node中的表现

      • 实现方式:
        • 表现为双工流对象
        • 继承自stream.Duplex类
      • 特殊性质:
        • 可同时进行读写操作
        • 与底层socket文件保持连接

    • 通过socket进行数据传输

      • 数据发送:
        • 通过向流写入内容实现
        • 示例：socket.write(“数据内容”)
      • 数据接收:
        • 通过监听’data’事件获取
        • 示例：socket.on(“data”, chunk=>{…})

    • 示例：建立socket连接并接收服务器消息

      • 完整流程:

    • 注意事项:

      • 需要主动发送请求才会收到响应
      • 默认不会自动发送HTTP请求头

    • TCP/IP连接中的客户端与服务器交互

      • 交互机制:
        • 三次握手建立连接后，需要客户端主动发送请求
        • 仅建立连接通道不会触发服务器响应
      • HTTP协议差异:
        • TCP/IP层不包含自动请求机制
        • 需要手动实现HTTP协议的消息交换

  • socket.write

    • 写入内容类型: 可以向流中写入字符串或Buffer对象，例如UTF-8编码的文本数据
    • 实际应用示例: 通过socket.write(“你好”)向腾讯课堂服务器发送消息
    • 底层原理: 该方法本质是将数据写入TCP/IP协议的传输通道中

  • HTTP协议字符串格式

    • 响应结构:
      • 响应行: 包含协议版本和状态码，如HTTP/1.1 400 Bad Request
      • 响应头: 包含服务器信息、日期等元数据，如Server: stgw/1.3.12.3_1.13.5
      • 响应体: 包含实际返回内容，通常是HTML格式的错误页面
    • 协议规范:
      • 正确格式要求: 必须包含请求行、请求头(换行分隔)、请求体(双换行分隔)
      • 错误示例分析: 发送简单字符串”你好”不符合HTTP协议格式，导致400错误
    • 协议对比:
      • HTTP与TCP区别: 普通TCP服务器能识别任意格式数据，而HTTP服务器有严格格式要求
      • 底层本质: HTTP响应本质是特定格式的字符串，客户端库会将其解析为对象
    • 格式细节:
      • 换行要求: 必须使用换行符(不能含多余空格或Tab)分隔各部分
      • 请求示例: GET请求不需要请求体，POST请求需要包含请求体
    • 开发启示:
      • 错误处理: 400错误表明请求格式不符合HTTP协议规范
      • 调试技巧: 可通过查看原始响应字符串验证协议格式是否正确

  • GET和POST请求表示

    • HTTP请求分析

      • 协议层级关系：HTTP协议建立在TCP/IP协议之上，TCP/IP协议负责建立连接和传输数据，HTTP协议规定通信模式（请求-响应）和消息格式
      • 请求方法本质：GET/POST等HTTP方法是通过请求行中的文本标识的，不是TCP/IP协议的属性。例如请求行格式为：GET / HTTP/1.1GET\ /\ HTTP/1.1GET / HTTP**/1.1**
      • 请求体分隔规则：请求头与请求体之间必须有两个换行符作为间隔，即使没有请求体也需要保留空行
      • 请求组成要素：

        • 请求行：包含方法、路径和协议版本（如GET / HTTP/1.1GET\ /\ HTTP/1.1GET / HTTP**/1.1**

          ）

        • 请求头：包含Host等元信息（如Host: duyi.ke.qq.comHost:\ duyi.ke.qq.comHost**:** duyi**.ke**.qq.com

          ）

        • 请求体：POST请求时携带的数据（GET请求通常为空）

      • 底层实现细节：
        • 连接保持：通过Connection:keep-alive头可维持TCP连接
        • 消息传输：TCP连接建立后，客户端和服务器可以双向传输消息
        • 终止条件：HTTP/1.1默认不会立即关闭连接，服务器可能保持连接等待后续请求
      • 响应格式解析：

        • 状态行：包含协议版本和状态码（如HTTP/1.1 200 OKHTTP/1.1\ 200\ OKHTTP**/1.1** 200 OK

          ）

        • 响应头：包含Content-Type等元信息

        • 响应体：实际的HTML文档或其他资源内容

      • 工具使用建议：
        • Postman：用于测试HTTP请求的专业工具，可查看原始请求格式
        • 网络调试：通过Node.js的net模块可直接操作TCP连接，观察原始HTTP报文

  • socket.close

    • 关闭触发条件：当客户端调用socket.end()或socket.close()方法时触发关闭事件
    • 双向关闭特性：客户端主动关闭连接后，服务器端会自动同步关闭对应socket连接
    • 长连接处理：非长连接情况下，客户端应在接收完服务器响应后立即调用关闭方法

  • 判定传输完成

    • 流式传输特性

      • 分块机制：TCP流数据以分块(chunk)形式传输，单次data事件无法保证接收完整消息
      • 缓冲区限制：受网络MTU和缓冲区大小限制，大响应会被自动拆分为多个数据包

    • Content-Length判定法

      • 关键字段：响应头中的Content-Length字段指示消息体总字节数（示例中为60358字节）
      • 累计接收原理：需要持续累加各chunk的Buffer.byteLength直到达到指定长度
      • 头体分离：通过首个\n\n分割响应头和消息体，头部信息可立即用字符串处理

  • parseHeader

    • 实现原理

      • 分割标识：使用indexOf(“\n\n”)定位头体分界位置
      • 字符串截取：通过substring(0, index)提取纯头部内容
      • 多块处理：只需处理首个chunk中的头部信息，后续chunk均为消息体

    • 注意事项

      • 多次触发：data事件会因分块传输多次触发，需要维护接收状态
      • 编码处理：必须统一使用utf-8编码进行字符串转换
      • 性能优化：避免在每次data事件中都进行完整字符串转换

  • parseResponse

    • 响应头和响应体分割示例

      • 分割原理：通过”\n\n”或”\r\n\r\n”来分割HTTP响应的消息头和消息体
      • 实现方法：
        • 使用indexOf(“\n\n”)查找分割位置
        • head部分使用substring(0, index)获取
        • body部分使用substring(index + 2)获取（跳过两个换行符）
      • 注意事项：
        • 需要设置isFirst标志位，只在第一次接收数据时进行解析
        • 后续数据块可能也包含”\n\n”，但不应重复解析

    • 响应头和响应体对象处理

      • 响应头处理流程：
        • 使用split(“\r\n”)将头部分割成数组
        • 使用slice(1)跳过第一行状态行
        • 每行用split(“:”)分割键值对，并用trim()去除空格
      • 对象转换：
        • 使用reduce将键值对数组转换为对象
        • 属性名为键，属性值为值
      • body处理：
        • 使用trimStart()去除开头空白字符
        • 可通过Buffer.from(body).byteLength获取接收字节数

    • 接收控制：

      • 通过content-length判断是否接收完成
      • 比较已接收字节数和总字节数确定是否结束

    • 错误处理：

      • 注意reduce初始值必须返回新对象
      • 键名可能包含特殊字符，需要正确处理

二、接收字节判断逻辑

1. 字节数比较机制
  • 结束条件判断：当接收到的字节数大于等于总字节数时，表示传输结束（is over）。这是HTTP协议中通过Content-Length判断请求结束的核心机制。
  • 首次接收处理：
    • 若第一次接收就完成全部数据（socket触发end事件），直接结束方法
    • 若第一次未接收完，仅完成对象解析，不做其他处理
  • 后续接收流程：
    • 未结束时持续将chunk拼接到response.body
    • 每次拼接后重新检查is over条件
    • 最终在socket close事件输出完整的receive.body
2. 实现细节
  • 数据拼接时机：非首次接收时立即执行body拼接操作
  • 结束事件处理：在socket的end和close事件中完成最终处理
  • 边界情况：需考虑结尾字符可能存在的计数偏差（实际案例中出现多1字节情况）

三、服务器创建

1. 服务器基础
  • 创建方法：使用createServer()创建，无需参数
  • 监听端口：通过listen(port)方法启动监听（如9527端口）
  • 运行特性：服务器进程会持续运行等待客户端连接
2. 核心事件
  • listening事件：
    • 触发时机：服务器成功绑定端口后
    • 典型用途：记录日志或显示服务启动提示
  • connection事件：
    • 触发条件：客户端成功建立连接时
    • 回调参数：包含通信用的socket对象
    • 连接特性：单个客户端可能产生多个连接（如浏览器预连接）
3. 通信模型
  • 多连接处理：
    • 每个新连接都会生成独立的socket
    • 类比客服系统：每个来电分配专属接线员
  • socket使用：
    • 写入操作：向客户端发送响应
    • 读取操作：获取客户端请求
    • 生命周期：短连接立即销毁，长连接超时后销毁

四、服务器响应客户端请求

1. Socket事件处理
  • on close事件：当连接关闭时触发，可以在事件处理函数中输出”连接关闭了”的提示信息。
  • on end事件：表示数据传输结束，但需要注意连接可能不会立即关闭。
2. 连接状态管理
  • 连接保持机制：如果服务器没有发送数据给客户端，连接会一直保持而不会关闭。
  • 主动断开连接：服务器可以通过调用socket.end()方法主动断开连接。
3. HTTP响应格式
  • 正确响应结构：
    • 协议和状态码（如HTTP/1.1 200 OK）
    • 响应头（如Content-Type）
    • 两个换行符
    • 响应体（如HTML内容）
  • 常见错误：响应格式不符合HTTP协议规范会导致浏览器无法正确解析。
4. 内容类型设置
  • Content-Type作用：告诉浏览器如何解析响应内容。
    • text/plain：纯文本显示
    • text/html：HTML解析
    • image/jpeg：图片解析
  • 示例：设置Content-Type: text/html后，浏览器会将响应体作为HTML渲染。
5. 文件响应实现
  • 文件读取：使用fs.readFile读取文件内容（如图片）。
  • Buffer处理：
    • 将响应头和文件内容分别转换为Buffer
    • 使用Buffer.concat()合并多个Buffer
  • 完整流程：
    • 接收客户端请求
    • 读取文件内容
    • 构建正确格式的响应头
    • 合并响应头和文件内容
    • 发送响应并结束连接
6. 注意事项
  • 格式严格性：HTTP响应中不能有多余的空格或字符。
  • 写入顺序：确保所有数据写入完成后再调用end()方法。
  • 性能考虑：直接使用net模块创建HTTP服务器较为繁琐，实际开发中通常使用更高级的模块。
7. HTTP协议特点
  • 连接管理：HTTP协议通常会在响应完成后断开连接。
  • 请求格式：包含请求行、请求头和请求体三部分。
  • 响应格式：包含状态行、响应头和响应体三部分。
