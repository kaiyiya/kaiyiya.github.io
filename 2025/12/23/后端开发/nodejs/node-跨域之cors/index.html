node-跨域之CORS

飞向月球的猫2025-12-232025-12-23

一、CORS原理详解

1. 概述
1）JSONP原理及其问题
  • 原理：JSONP通过动态创建script标签实现跨域请求，要求服务器返回JS代码而非JSON数据。
  • 问题1-消息格式混乱：服务器需要针对跨域和非跨域请求返回不同格式（JS代码/JSON），增加服务器复杂度。
  • 问题2-仅支持GET：由于script标签限制，JSONP只能发起GET请求，无法使用POST等其他HTTP方法。
  • 替代方案：CORS是更优的跨域解决方案，除非维护非常老的系统。
2）CORS的原理与总体思路

  • 全称：Cross-Origin Resource Sharing（跨域资源共享）
  • 核心思想：浏览器向服务器声明跨域请求来源(Origin)，服务器通过响应头明确授权。
  • 交互流程：
    • 浏览器添加Origin头标识请求来源
    • 服务器响应Access-Control-Allow-Origin头
    • 浏览器验证通过后交付响应数据
  • 安全机制：基于HTTP1.1协议，是对同源策略的安全补充。
3）CORS的三种交互模式

  • 分类依据：根据请求对服务器的影响程度划分
  • 模式演进：
    • 简单请求：只读操作，如获取新闻
    • 需预检请求：可能修改数据的操作
    • 带凭证请求：涉及身份认证的操作
  • 安全设计：越高级的模式要求越严格，形成安全梯度。
2. 简单请求
1）简单请求的判定

  • 条件1-方法限制：必须是GET/POST/HEAD方法之一
    • HEAD请求用于获取资源元信息，不返回消息体
  • 条件2-安全头字段：仅允许以下请求头：
    • Accept、Accept-Language、Content-Language等标准头
    • 禁止自定义头字段
  • 条件3-Content-Type限制：若存在只能是：
    • text/plain
    • multipart/form-data
    • application/x-www-form-urlencoded
  • 示例分析：
    • fetch(“http://crossdomain.com/api/news")：简单请求
    • 添加额外头字段或使用PUT方法：非简单请求
2）简单请求的交互规范
  • 请求阶段：
    • 浏览器自动添加Origin头（如Origin: http://my.com）
    • 示例请求头包含：Host、Connection、Referer等标准字段
  • 响应阶段：
    • 服务器必须包含Access-Control-Allow-Origin响应头
    • 值可以是：
      • *：允许所有源
      • 具体源（如http://my.com）：更安全推荐
  • 实现建议：
    • 服务器维护允许源列表进行动态校验
    • 避免使用*，防止后续凭证请求出现问题
  • 调试技巧：
    • 文件协议打开的页面Origin为null
    • 开发服务器需配置允许本地地址（如127.0.0.1:5500）
3. 需要预检的请求

1）浏览器发送预检请求
  • 触发条件：当请求不满足简单请求的三个条件时，浏览器会自动发送预检请求
  • 请求特征：
    • 方法为OPTIONS
    • 没有请求体
    • 包含特殊请求头：
      • Origin：请求源
      • Access-Control-Request-Method：后续真实请求方法
      • Access-Control-Request-Headers：后续会改动的请求头
  • 示例代码：
2）服务器允许

  • 响应要求：
    • 状态码200 OK
    • 必须包含的响应头：
      • Access-Control-Allow-Origin：允许的源
      • Access-Control-Allow-Methods：允许的方法
      • Access-Control-Allow-Headers：允许的请求头
    • 可选响应头：
      • Access-Control-Max-Age：预检缓存时间（秒）
  • 缓存机制：通过Access-Control-Max-Age设置缓存时间，在有效期内相同请求不再发送预检
3）浏览器发送真实请求

  • 请求特征：
    • 包含原始请求的所有内容
    • 自动带上Origin头
    • 与简单请求处理流程相同
4）服务器响应真实请求
  • 响应要求：
    • 必须包含Access-Control-Allow-Origin头
    • 响应体为实际业务数据
    • 即使有缓存仍需验证权限
  • 完整流程：预检请求→预检响应→真实请求→真实响应
4. 附带身份凭证的请求

  • 默认行为：默认情况下，AJAX跨域请求不会附带cookie，导致需要权限的操作无法进行。

  • 客户端配置方法：

    • XMLHttpRequest：设置xhr.withCredentials = true
    • Fetch API：配置fetch(url, {credentials: “include”})

  • 请求头变化：当附带cookie时，无论简单请求还是预检请求，都会在请求头中添加cookie字段。

  • 服务器响应要求：

    • 必须明确在响应头中添加Access-Control-Allow-Credentials: true
    • 禁止设置Access-Control-Allow-Origin: *，必须指定具体域名

  • 严格性说明：若服务器未明确允许凭证，即使客户端已配置携带cookie，浏览器仍会拒绝跨域请求。

  • 服务器实现：

  • credentials参数选项：

    • omit：始终不带cookie
    • same-origin：仅同域时携带（默认值）
    • include：始终携带cookie
5. 一个额外的补充
  • 默认限制：跨域时JS只能访问基础响应头（如Cache-Control、Content-Type等）
  • 白名单配置：
    • 通过Access-Control-Expose-Headers指定可访问的额外响应头
    • 示例：Access-Control-Expose-Headers: authorization, a, b
  • 实现意义：允许客户端获取服务器自定义的特殊头信息，如认证令牌等。
  • 预检请求处理：需要响应OPTIONS方法并设置对应头信息
  • 安全注意事项：
    • 必须维护允许的origin白名单（如[“http://127.0.0.1:5500“, “null”]）
    • 生产环境应避免使用通配符*
    • 需处理Chrome的SameSite cookie策略警告

二、知识小结

知识点 核心内容 考试重点/易混淆点 难度系数
CORS跨域方案 基于HTTP 1.1的跨域资源共享机制，通过服务器响应头（如Access-Control-Allow-Origin）控制跨域权限 与JSONP对比：JSONP仅支持GET请求且需服务器返回JS代码，CORS支持所有HTTP方法 ⭐⭐⭐
简单请求判定条件 1. 请求方法为GET/POST/HEAD2. 请求头仅含安全字段（如Accept、Content-Language）3.Content-Type仅限text/plain、multipart/form-data、application/x-www-form-urlencoded 易混淆点：Content-Type为application/json时即非简单请求 ⭐⭐
预检请求（Preflight） 非简单请求时，浏览器先发OPTIONS请求询问服务器权限，需服务器响应Access-Control-Allow-Methods和Access-Control-Allow-Headers 关键头字段：Access-Control-Request-Method（声明真实请求方法）、Access-Control-Max-Age（缓存预检结果时间） ⭐⭐⭐⭐
附带凭证的请求 需设置credentials: ‘include’（Fetch API）或withCredentials: true（XHR），服务器需响应Access-Control-Allow-Credentials: true 限制：Access-Control-Allow-Origin不能为*，必须指定具体域名 ⭐⭐⭐⭐
响应头扩展权限 通过Access-Control-Expose-Headers允许客户端读取额外响应头（如自定义头字段） 典型场景：前端需获取X-Token等自定义头信息 ⭐⭐
