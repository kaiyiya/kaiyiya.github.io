一、JWT原理详解

1. 大纲
1）JWT出现的原因

  • 背景需求：随着前后端分离架构的普及和数据中心的建立，公司需要为多种终端设备（浏览器、移动端、桌面应用等）提供统一认证方案。
  • 传统方案局限：传统cookie机制主要针对浏览器设计，其他终端设备缺乏完善的cookie管理机制，需要手动处理安全性问题。
2）中心服务器出现的必要性

  • 产品线特性：不同产品线通常有独立服务器处理内部数据交互，但需要共享核心数据（如用户数据）。
  • 典型案例：腾讯QQ、微信、邮箱等产品共用用户体系，实现单点登录功能。
  • 核心功能：中心服务器至少需要承担认证授权功能，统一管理跨产品线的身份验证。
3）中心服务器承担的功能
  • 核心职责：接收各终端设备的认证请求，响应身份令牌（如JWT）。
  • 通信方式：通过HTTP协议与各类终端设备进行通信，不限定具体终端类型。
2. 概述
1）浏览器和其他设备对cookie的处理

  • 技术可行性：所有设备理论上都可以处理cookie（本质是HTTP消息头），但浏览器有自动管理机制。
  • 实际限制：非浏览器设备需要开发者手动实现cookie存储、安全策略等完整机制，存在实现复杂度和安全隐患。
2）JWT的出现及作用
  • 解决方案：JWT提供统一的令牌格式，解决多终端认证标准化问题。
  • 设计目标：建立跨平台的、安全的令牌传输规范，不依赖特定终端特性。
3）JWT的全称及定义

  • 术语解析：全称JSON Web Token，可理解为”JSON格式的互联网令牌”。
  • 本质特征：是一种标准化的令牌数据格式，与传输方式、存储位置无关。
4）JWT的存储与传输方式

  • 存储自由：可存储在cookie、localStorage、手机文件、PC文件等任意位置。
  • 传输自由：可通过HTTP头、消息体等任意方式传输，常见于Authorization头。
5）JWT令牌在响应中的位置
  • 响应位置：可出现在Set-Cookie头、自定义头（如Authorization）或响应体中。
  • 混合方案：可同时在多个位置返回，兼顾浏览器便利性与其他终端需求。
6）JWT令牌的存储
  • 客户端职责：获取令牌后只需确保安全存储，具体存储方式根据终端特性决定。
  • 浏览器方案：推荐localStorage或cookie；原生应用可使用系统安全存储。
7）JWT令牌的发送与验证

  • 请求携带：通常采用Authorization: Bearer <token>格式（OAuth2规范）。
  • 验证流程：服务器收到令牌后验证有效性，整个过程与cookie方案类似但更通用。
8）JWT令牌的组成
  • 头部(header)：记录令牌类型和签名算法（如HS256）。
  • 负载(payload)：包含主体信息（如用户ID、权限等业务数据）。
  • 签名(signature)：用于验证令牌完整性和真实性。
3. 令牌的组成
  • 规范要求：JWT是一个开放标准(RFC 7519)，规定了令牌的固定格式，具体实现库可以自由选择
  • 三部分结构：由Header(头部)、Payload(负荷)、Signature(签名)三部分组成，组合格式为
    header.payload.signatureheader.payload.signatureheader.payload.signature
1）header

  • 核心功能：记录令牌类型和签名算法
  • JSON结构：
  • 算法说明：
    • alg：签名部分使用的加密算法，常用HS256(对称加密)或RS256(非对称加密)
    • typ：固定值”JWT”，标识令牌类型
  • 编码方式：
    • 使用base64url编码（对base64中+、=、/三个特殊字符处理）
    • 浏览器可用btoa()编码/atob()解码
    • Node.js需第三方库实现
2）payload

  • 数据载体：存储主体信息，采用JSON格式
  • 标准字段(可选)：
    • iss：签发者标识
    • iat：签发时间戳(秒)
    • exp：过期时间戳
    • sub：令牌用途说明
    • aud：目标接收方
    • nbf：生效时间戳
    • jti：唯一编号(防重放攻击)
  • 自定义字段：可自由添加任意字段（如用户ID、账号等）
  • 编码方式：同header使用base64url编码
3）signature
  • 安全机制：防止令牌伪造和篡改
  • 生成方式：
    • 拼接编码后的header和payload：
      encodedHeader+”.”+encodedPayloadencodedHeader + “.” + encodedPayloadencodedHeader**+“.”+encodedPayloa**d
    • 使用header指定的算法(如HS256)和服务器密钥进行加密
  • 验证原理：
    • 客户端无法伪造：缺少服务器密钥
    • 防篡改：修改payload会导致签名验证失败
  • 传输安全：
    • header和payload仅编码未加密（相当于明文）
    • 敏感信息不应存入payload
    • 必要时可对整个令牌二次加密
4. 令牌的验证
1）验证流程

  • 验证原理：服务器通过重新计算签名来验证令牌是否被篡改。具体步骤：
    • 从传入的JWT中提取header和payload
    • 使用相同的密钥和header中指定的加密算法重新计算签名
    • 将新计算的签名与传入的signature进行比对
  • 验证公式：
    新的signature=header中的加密算法(传入的header+传入的payload,秘钥)新的signature = header中的加密算法(传入的header+传入的payload, 秘钥)新的signature**=header中的加密算法**(传入的header**+传入的payload**,秘钥)
  • 验证结果：
    • 如果
      新的signature==传入的signature新的signature == 传入的signature新的signature**==传入的signature**
      ，则验证通过
    • 如果不一致，则说明令牌被篡改过
2）其他验证
  • 验证特点：
    • 手动验证：所有验证都需要服务器手动完成，没有自动验证机制
    • 可扩展性：可以借助第三方库简化验证过程
  • 常见验证项：
    • 过期时间：检查payload中的exp字段是否过期
    • 听众验证：验证aud(受众)是否符合要求
    • 签发者：验证iss(签发者)是否可信
  • 注意事项：
    • payload格式可以自定义，但验证逻辑需要自行实现
    • 篡改检测是基础验证，其他业务验证需要在此基础上进行
5. 总结
1）JWT特点
  • 本质特性：
    • 格式规范：本质是一种令牌格式规范，与终端设备、服务器技术、传输方式无关
    • 三部分结构：由header、payload、signature三部分组成，主体信息存储在payload中
  • 安全特性：
    • 防篡改：通过签名机制(signature)确保难以被篡改和伪造
    • 明文传输：header和payload是Base64编码的明文，但不可随意修改
2）应用场景

  • 中心化认证：
    • 场景需求：随着前后端分离和数据中心建设，需要统一的认证中心
    • 典型架构：中心服务器处理认证授权，各产品线服务器处理业务逻辑
  • 跨终端支持：
    • 设备兼容：支持浏览器、移动端、桌面应用、智能家居等多种终端
    • 共享数据：特别是用户数据等需要在各产品线间共享的信息
  • 通信方式：
    • 基于HTTP协议进行通信
    • 身份令牌替代传统的cookie机制

二、知识小结

知识点 核心内容 考试重点/易混淆点 难度系数
JWT定义与作用 JSON格式的互联网令牌，解决多终端统一认证问题 与Cookie的区别：JWT是规范格式，Cookie是浏览器存储机制 ⭐⭐
JWT组成结构 三部分：Header(算法类型)、Payload(主体数据)、Signature(数字签名) Base64URL编码不是加密，只是传输编码方式 ⭐⭐⭐
Header部分 记录算法(HS256/RS256)和令牌类型(jwt) 算法选择决定加密方式(对称/非对称) ⭐⭐
Payload部分 可自定义字段，规范建议字段(iss/sub/exp等) 明文传输但需防篡改，敏感数据避免存放 ⭐⭐⭐⭐
Signature机制 用密钥对前两部分加密生成，防止伪造和篡改 密钥保管是安全核心，客户端无法伪造 ⭐⭐⭐⭐
令牌验证流程 服务器重新计算签名比对，检查过期时间等 需手动实现全部验证逻辑 ⭐⭐⭐
传输方式 可放Cookie/Header/URL等，与传输媒介解耦 Authorization头是常见方式但非强制 ⭐⭐
应用场景 前后端分离、多终端单点登录、中心化认证 比Cookie更适合非浏览器环境 ⭐⭐⭐
安全特性 防篡改靠签名，防伪造靠密钥，不解决信息泄露 敏感操作仍需二次验证 ⭐⭐⭐⭐
