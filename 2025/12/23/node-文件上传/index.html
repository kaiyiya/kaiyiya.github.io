一、文件上传

1. 文件上传原理
1）百度图片上传接口

  • 消息体格式选择

    • 特殊格式说明：图片上传需要使用form-data格式而非JSON或x-www-form-urlencoded格式
    • 服务器兼容性：虽然理论上GET请求也可以上传图片（带有消息体），但大多数服务器不会处理GET请求中的文件数据
    • 实际应用：演示中使用POST请求向百度服务器上传图片，虽然百度不会处理但数据确实已发送

  • multipart/form-data格式详解

    • Content-Type：必须设置为multipart/form-data并包含boundary参数
    • boundary作用：作为分隔符标记表单中不同数据段的边界，格式为”–WebKitFormBoundary”加随机字符串
    • 数据格式示例：

  • 文件上传实践

    • 参数设置：
      • 文本参数：键值对形式（如a=1，b=2）
      • 文件参数：需要指定字段名和选择本地文件
    • 多文件上传：可以同时添加多个文件字段，每个字段对应一个文件
    • 文件大小说明：图片文件通常较大（至少1-2KB），1KB相当于1024个英文字符的数据量

  • 服务器文件名处理

    • 客户端文件名作用：主要供服务器参考文件后缀名，实际存储不应直接使用
    • 重名问题：不同用户上传同名文件会导致覆盖，示例说明客户端1和客户端2都上传”吃瓜.gif”的冲突情况
    • 解决方案：服务器应生成唯一文件名保存，避免不同用户的文件相互覆盖

  • 请求响应处理

    • 响应要求：服务器处理完成后需要返回文件存储路径供客户端访问
    • 代码生成：Postman可自动生成各种语言的请求代码（如JavaScript Fetch API实现）
    • 实际应用：虽然演示中百度返回”页面不存在”，但验证了请求格式的正确性和数据完整性
2）传统表单图片上传实现

  • 表单基础结构

    • 基本元素：使用 <form>标签创建表单，必须设置三个关键属性：
      • action=”/api/upload”：指定提交到服务器的接口地址
      • method=”POST”：使用POST请求方式
      • enctype=”multipart/form-data”：设置表单编码类型为多部分表单数据

  • 服务器端处理

    • 路由创建：
      • 在routes/api/upload.js中创建处理路由
      • 使用Express的Router处理POST请求
    • 基础响应：
    • 路由注册：

  • 请求过程分析

    • 请求头特征：
      • Content-Type: multipart/form-data：标识表单数据类型
      • 自动生成boundary分隔符（如WebKitFormBoundary4bCPwrNuzEHmAN75）
    • 传统方式特点：
      • 页面会刷新跳转
      • 服务器响应直接显示在页面上
      • 适合简单的文件上传场景
    • 注意事项：
      • 必须正确设置表单的enctype属性
      • 服务器路由需要正确导出和注册
      • 文件字段需要使用 <input type="file">

  • 与传统AJAX方式的对比

    • 主要区别：
      • 传统表单会刷新页面，AJAX可以实现无刷新上传
      • AJAX可以更灵活地处理响应结果
      • 传统方式浏览器会自动处理请求体构造，AJAX需要手动处理
    • 适用场景：
      • 传统方式适合简单的表单提交
      • AJAX方式适合需要动态交互的现代Web应用
3）Ajax方式图片上传

  • 前端实现原理

    • 表单元素：包含文本输入框(input type=”text”)和文件上传框(input type=”file”)，通过name属性标识
    • 事件绑定：通过document.querySelector获取按钮元素，设置onclick事件调用upload函数
    • DOM操作：使用querySelector获取表单元素值，如document.querySelector(“[name=a]”)

  • FormData对象使用

    • 作用：自动构建form-data格式的消息体，避免手动处理分隔符
    • API方法：
      • append()：添加键值对，支持三种参数：字段名、字段值、文件名(可选)
      • 示例：formData.append(“a”, inpA.value)添加文本字段
    • 文件处理：
      • 通过files属性获取文件列表(FileList对象)
      • 支持多文件上传：添加multiple属性 <input type="file" multiple>
      • 循环添加多个文件：for (const file of inpFile.files)

  • Fetch API上传实现

    • 请求配置：
      • method必须设置为”POST”
      • body直接传入FormData对象
    • 响应处理：
      • 使用.then(resp => resp.text())将响应解析为文本
      • 通过console.log输出响应结果
    • 网络观察：
      • Content-Type自动设置为multipart/form-data
      • 消息体包含分隔符和文件二进制数据

  • 后端处理注意事项

    • 中间件限制：
      • express.urlencoded和express.json无法解析multipart/form-data格式
      • 需要专门的文件上传中间件(如multer)或自定义解析逻辑
    • 解析原理：
      • 按分隔符(–WebKitFormBoundary…)分段读取请求体
      • 识别Content-Disposition头获取字段名和文件名
      • 处理二进制文件数据时需要特殊处理

  • 开发调试技巧

    • 浏览器工具：
      • 使用Network面板查看实际请求头和消息体
      • 通过Console面板调试文件选择结果
    • 文件对象属性：
      • name：文件名
      • size：文件大小(字节)
      • type：MIME类型
      • lastModified：最后修改时间戳
4）文件上传服务器解析处理请求体

  • Multer介绍

    • 核心功能：Multer是Node.js中间件，专门处理multipart/form-data类型的表单数据，主要用于文件上传

    • 底层实现：基于busboy构建，具有高效的文件处理能力

    • 限制条件：只能处理multipart/form-data类型的表单，其他类型表单会被忽略

    • 安装方式：通过npm安装，命令为$npm install –save multer

    • 请求对象扩展：

      • 添加body对象：存储表单文本域信息（如a=1,b=2）
      • 添加file/files对象：存储上传的文件信息

    • 文件存储方式：

      • 默认将文件保存在内存中
      • 可通过dest配置指定磁盘存储路径

    • 安全建议：不应作为全局中间件使用，只应在需要处理文件上传的路由中使用

  • Multer配置与使用

    • 三种处理方式：

      • .single(fieldname)：处理单个文件，文件信息保存在req.file
      • .array(fieldname[, maxCount])：处理多个文件（同名字段），文件信息保存在req.files数组
      • .fields(fields)：处理多个文件（不同名字段），文件信息保存在req.files对象

    • 文本域处理：

      • .none()：仅处理文本域，若收到文件会报错
      • .any()：接受所有上传文件（慎用）

    • 文件对象属性：

      • fieldname：表单指定的字段名
      • originalname：用户原始文件名
      • encoding：文件编码
      • mimetype：文件MIME类型
      • size：文件大小（字节）
      • destination：存储路径（DiskStorage）
      • filename：存储文件名（DiskStorage）
      • path：完整文件路径（DiskStorage）
      • buffer：文件Buffer（MemoryStorage）

  • 配置文件存储路径

    • dest配置：

      • 最基本配置项，指定文件存储目录
      • 示例：dest: path.resolve(__dirname, “../../public/upload”)
      • 目录不存在时会自动创建

    • 注意事项：

      • 未配置dest时文件保存在内存中
      • 上传文件会被重命名以避免冲突
      • 生产环境应配置.gitignore忽略上传目录

    • 其他配置项：

      • storage：替代dest的更高级存储配置
      • fileFilter：文件类型过滤器
      • limits：限制上传数据大小
      • preservePath：保留完整文件路径

二、文件上传

1. 客户端文件上传
1）文件上传与后缀名问题
  • 默认处理机制：服务器默认不保存文件后缀名，这是为了安全考虑，因为需要处理各种类型的文件
  • 图片处理场景：在图片上传场景中需要保留后缀名，否则会导致浏览器无法正确识别文件类型
  • 解决方案：可以通过配置存储引擎来解决后缀名问题，确保文件能正确显示
2）客户端文件上传代码示例
  • FormData构建：
    • 使用new FormData()构建表单数据
    • 通过append()方法添加文本字段和文件字段
    • 文件字段需要循环处理inpFile.files数组
  • fetch请求：
    • 设置请求方法为POST
    • 请求体直接传入FormData对象
    • 处理响应结果时使用.text()方法获取文本响应
3）文件上传路径问题
  • 路径引用问题：
    • 上传后的文件存储在public/upload目录下
    • 通过 <img>标签的src属性引用上传文件
    • 无后缀名文件会导致浏览器无法正确识别文件类型
  • 验证方法：
    • 直接在页面中通过图片元素引用上传文件
    • 观察浏览器是否能正确显示图片内容
4）文件上传路径被修改的原因
  • history中间件影响：
    • 无后缀名文件请求会被重定向到index.html
    • 这是history中间件的默认行为
    • 临时解决方案是注释掉history中间件
  • 根本解决方案：
    • 必须保留文件后缀名
    • 确保浏览器能正确识别文件类型
    • 避免请求被错误重定向
5）磁盘存储引擎配置
  • 存储引擎类型：
    • 磁盘存储(DiskStorage)：文件保存到服务器磁盘
    • 内存存储：文件保存在内存中
    • 第三方存储：需要自行实现
  • 磁盘存储配置：
    • 通过multer.diskStorage()配置
    • destination设置存储目录
    • filename设置文件名生成规则
  • 安全警告：
    • 不要将multer作为全局中间件使用
    • 只在需要处理文件上传的路由中使用
    • 防止恶意用户上传文件到未受保护的路由
2. 服务器文件上传
1）upload.js源码：配置storage
  • 基本结构：使用multer.diskStorage()方法创建存储引擎，包含destination和filename两个关键配置项
  • 简写方式：当变量名相同时可采用ES6属性简写，如multer({ storage })
  • 存储控制：相比直接使用dest参数，storage配置能更精细控制文件存储位置和命名规则
2）磁盘存储引擎：destination和filename

  • destination函数：
    • 作用：确定文件存储目录，接收req, file, cb三个参数
    • 创建责任：使用函数形式时需要手动创建目录，字符串形式则由multer自动创建
    • 默认值：未设置时使用系统临时文件夹
  • filename函数：
    • 命名规则：需返回完整文件名（含扩展名），multer不会自动添加扩展名
    • 默认行为：未设置时生成随机无扩展名文件名
    • 参数说明：req.body可能未完全填充，需注意字段顺序影响
3）upload.js源码：destination设置
  • 路径构建：
    • 方法：使用path.resolve(__dirname, “../public/upload”)构建绝对路径
    • 层级关系：通过../返回上级目录，再定位到public/upload子目录
  • 回调规范：
    • 错误处理：第一个参数传null表示无错误
    • 路径返回：第二个参数传递最终存储路径字符串
  • 注意事项：需确保目标目录存在，否则需手动创建
4）upload.js源码：构建绝对路径
  • 路径解析：
    • __dirname表示当前模块目录
    • path.resolve()方法将相对路径解析为绝对路径
  • 目录结构：
    • 示例中采用”../../“跨越两级目录
    • 最终定位到项目根目录下的public/upload文件夹
5）文件信息

  • 核心字段：
    • fieldname：表单中定义的字段名
    • originalname：客户端原始文件名
    • mimetype：文件MIME类型（如image/jpeg）
    • size：文件字节大小
  • 存储相关：
    • destination：磁盘存储目录（仅DiskStorage）
    • filename：存储后的文件名
    • path：文件完整存储路径
6）upload.js源码：filename设置
  • 基础方案：
    • 组合file.fieldname和Date.now()生成文件名
    • 示例：file.fieldname + “-“ + Date.now()
  • 扩展信息：
    • 可通过file.originalname获取客户端原始文件名
    • 使用path.extname()提取文件扩展名
7）文件名构建：时间戳+随机字符
  • 命名策略：
    • 时间戳：使用Date.now()获取毫秒级时间戳
    • 随机串：Math.random().toString(36).slice(-6)生成6位随机字符
    • 扩展名：从file.originalname提取原始扩展名
  • 完整示例：
  • 响应处理：
    • 返回JSON格式响应包含文件访问路径
    • 路径格式：/upload/生成的文件名
3. 文件上传配置
1）文件大小限制配置
  • limits配置项：用于限制上传文件的各项参数，包括文件名长度、字段值长度等
  • fileSize设置：
    • 默认值为infinity（无限制）
    • 需设置为整数，单位是字节
    • 示例：限制为10KB可设置为10*1024
  • 错误处理：
    • 超过限制会抛出MulterError
    • 错误消息为”file too large”
    • 建议在服务器端和客户端都进行验证
2）文件类型过滤
  • fileFilter配置：
    • 接收req和file对象作为参数
    • 通过判断file.originalname获取文件扩展名
  • 白名单机制：
    • 可设置允许的后缀名数组（如[“.jpg”, “.gif”, “.png”]）
    • 不在白名单则抛出错误信息”your extname is not support”
  • 客户端验证：
    • 可通过input元素的accept属性限制（如accept=”image/*”）
    • 但服务器端验证是必须的，客户端验证仅用于提升用户体验
3）错误处理机制
  • 错误类型判断：
    • 使用err instanceof MulterError识别上传错误
    • 可获取err.message获取具体错误信息
  • 响应处理：
    • 建议返回200状态码配合错误信息
    • 错误信息格式示例：{code:500, message:”file too large”}
  • 中间件处理：
    • 可在Express中间件中专门处理Multer错误
    • 灵活处理不同错误类型的响应方式
