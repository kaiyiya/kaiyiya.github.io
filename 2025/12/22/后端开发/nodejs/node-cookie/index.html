一、cookie的基本概念

1. 一个不大不小的问题
1）cookie的基本概念

  • 令牌机制：服务器在客户端登录成功后颁发令牌(token)，后续请求必须附带该令牌作为身份凭证
  • 现实类比：类似于政府部门”认证不认人”的办事方式，通过证件识别身份而非人脸识别
  • 技术本质：cookie是浏览器端存储的小型文本文件，单个网站cookie大小通常不超过4KB
2）http协议的无状态问题
  • 核心问题：HTTP协议无状态特性导致服务器无法识别连续请求是否来自同一客户端
  • 形象比喻：将服务器比作有严重脸盲症的”东哥”，无法记住客户端的历史交互记录
  • 典型场景：添加管理员等需要权限的操作时，服务器无法验证请求者的合法身份
3）服务器的认证机制
  • 解决方案流程：
    • 登录成功时服务器颁发令牌(出入证)
    • 后续请求自动附带该令牌
    • 服务器验证令牌有效性
  • 验证逻辑：令牌存在即证明之前通过认证，无需重复验证账号密码
  • 适用范围：仅需在敏感操作(如数据增删改)时启用认证，公开内容无需认证
4）cookie的功能与组成
  • 四大核心功能：
    • 多凭证存储：可保存多个网站/服务的令牌(如QQ邮箱和空间的独立令牌)
    • 自动出示：访问对应网站时自动附加相关cookie
    • 精准匹配：确保不会将A网站的cookie发送给B网站
    • 有效期管理：自动清理过期cookie
  • 扩展应用：
    • 存储用户偏好设置(如关闭教程提示的选项)
    • 记录广告展示状态等非认证信息
  • 技术实现：
    • 以键值对形式存储字符串数据
    • 每个cookie可设置独立属性(域名、路径、有效期等)
2. cookie的组成

  • 概念类比：cookie相当于浏览器的专属卡包，用于存放各个网站的身份信息（出入证/令牌token），具有自动管理功能
  • 核心功能：
    • 能够存放多个网站的出入证（一个网站可能有多个）
    • 自动出示对应网站的出入证（请求时附带）
    • 精确匹配出示（不会将A网站的cookie发给B网站）
    • 自动管理有效期（移除过期cookie）
1）键和值
  • 基本结构：每个cookie是一个键值对，相当于卡包中的一张卡片
  • 键(key)：标识信息的名称（如”身份编号”）
  • 值(value)：存储的具体信息（如”
    1456301550F2F76D69ECBF4BC951456301550F2F76D69ECBF4BC951456301550F2F76D69ECBF4BC95
    “）
  • 存储特点：
    • 可存储任何信息，但主要用于存储身份令牌
    • 相比local storage存储容量有限
    • 自动管理机制是其核心优势
2）域
  • domain属性：标识cookie所属的网站域名
  • 匹配规则：
    • 基础域名（如yuanjin.tech）可匹配所有子域名（www.yuanjin.tech、blogs.yuanjin.tech等）
    • 子域名（如www.yuanjin.tech）只能精确匹配该子域名
    • 不关心端口号，仅匹配域名
  • 示例：
    • cookie域为yuanjin.tech → 可匹配yuanjin.tech及其所有子域
    • cookie域为www.yuanjin.tech → 仅匹配www.yuanjin.tech
3）路径
  • path属性：标识cookie所属的网站路径（类似公司不同部门）
  • 匹配规则：
    • 指定路径（如/news）可匹配该路径及其子路径（/news/detail等）
    • 不能匹配其他路径（如/blogs）
    • 根路径（/）可匹配所有路径
  • 示例：
    • path为/news → 可匹配/news、/news/a/b/c
    • path为/ → 匹配所有请求路径
4）其他属性
  • secure属性：
    • true时：仅通过https协议传输
    • false时：http/https均可传输
  • expire属性：明确指定cookie过期时间
  • 自动管理机制：
    • 服务器颁发后浏览器自动保存
    • 过期自动移除
    • 请求时自动匹配并附加到请求头
5）发送机制
  • 发送条件（必须同时满足）：
    • 未过期（检查expire）
    • 域名匹配（检查domain）
    • 路径匹配（检查path）
    • 协议匹配（检查secure）
  • 请求头格式：
    • 多个cookie以键=值; 键=值形式拼接
    • 示例：Cookie: BIDUPSID=XXX; PSTM=1585383668; BD_UPN=123253
  • 安全警告：
    • cookie包含重要身份信息
    • 泄露可能导致他人冒用身份
    • 永远不要分享cookie内容
6）设置方式
  • 服务器设置：
    • 通过响应头Set-Cookie颁发
    • 最常见方式（如登录成功后颁发令牌）
  • 客户端设置：
    • 浏览器端JavaScript设置
    • 使用场景较少（如用户偏好设置）
  • 存储限制：
    • 单个cookie通常不超过4KB
    • 每个域名下的cookie数量有限制
3. 如何设置cookie
1）服务器端设置cookie

  • 基本原理: 当服务器决定给客户端颁发cookie时，会在HTTP响应消息中包含cookie信息，浏览器会自动保存到本地。

  • 基本格式: 每个cookie包含键值对和可选属性，格式为：
  • 必选/可选: 键值对是必须设置的，其他属性（path/domain/expire等）都是可选的，且顺序不限。
  • 覆盖规则: 当新cookie的key、path、domain与已有cookie完全相同时，会自动覆盖旧值。
2）客户端设置cookie
  • 实现方式: 通过JavaScript代码直接操作，如document.cookieAPI。
  • 典型应用:
    • 用户关闭广告时选择”不再显示”，JS将偏好设置保存到cookie
    • 早期没有localStorage时，本地存储主要依赖cookie
  • 交互流程: 客户端设置的cookie会在后续匹配的请求中自动发送给服务器
3）总结

  • 主要区别:

    • 服务器设置：更常见，用于身份认证等核心功能
    • 客户端设置：多用于用户偏好等非关键数据

  • 属性详解:

    • path：未设置时默认取当前请求路径（如请求/login则path为/login）
    • domain：未设置时默认取当前请求域
    • secure：仅限HTTPS请求发送
    • 过期控制：通过expire或max-age设置有效期

  • 实际案例: 访问百度首页时，服务器通过响应头设置多个cookie（如BDSVRTM、BD_HOME等），浏览器会自动保存并在后续请求中携带。

4. 响应示例百度的cookie设置
  • 示例观察：在百度响应头中可见多个Set-Cookie字段，如BDSVRTM=716; path=/和H_PS_PSSID=…; domain=.baidu.com，展示实际场景中的多cookie设置方式。
1）cookie的基本概念
  • 核心组成：每个cookie由键值对和可选属性组成，格式为键=值; path=?; domain=?; expire=?; max-age=?; secure; httponly。
  • 属性顺序：除键值对外，其他属性均为可选且顺序不限，如path和domain可调换位置。
2）cookie的组成
  • 唯一性判定：当key、path、domain三者完全相同时视为同一cookie，浏览器会自动覆盖旧值。
  • 覆盖机制：若已存在相同cookie卡片，新值会更新原有值、过期时间等属性。
3）cookie的路径设置
  • 默认行为：未显式设置path时，浏览器自动取当前请求路径作为默认值。例如请求/login时返回set-cookie:a=1，则path默认为/login。
  • 实际案例：访问百度首页(/)时，若响应cookie未指定path，则自动设为/；访问课程页(/course/231577)则path为/course/231577。
4）cookie的域设置
  • 默认取值：未设置domain时取请求域名，如请求http://www.yuanjin.tech则domain默认为www.yuanjin.tech。
  • 安全限制：浏览器会拒绝无效域设置，例如请求域为yuanjin.tech时，响应domain=baidu.com将被忽略，防止跨域安全风险。
5）cookie的过期时间
  • 绝对时间(expire)：必须为GMT格式字符串，如Fri,17 Apr 2020 09:35:59 GMT，可通过new Date().toGMTString()生成。
  • 相对时间(max-age)：单位为秒，优先级高于expire。设置max-age=1000时，浏览器自动计算当前时间+1000秒作为过期时间。
  • 会话周期：未设置expire和max-age时，cookie在浏览器窗口全部关闭后失效。
6）cookie的安全性设置
  • 传输限制：设置secure后仅允许HTTPS请求携带该cookie，普通HTTP请求不会发送。
  • 实际应用：百度示例中cookie未设置secure，故可通过HTTP和HTTPS两种协议传输。
7）cookie的HttpOnly属性
  • 防护机制：设置httponly后禁止客户端JS读取，有效防御XSS攻击窃取cookie。
  • 攻击示例：恶意iframe通过JS读取父页面cookie时，若cookie标记为httponly则读取失败。
5. 应用案例
1）例题：登录设置cookie示例

  • 登录流程：

    • 客户端通过POST请求发送账号密码到服务器
    • 服务器验证成功后，在响应头设置Set-Cookie字段
    • 示例响应头：set-cookie: token=123456; path=/; max-age=3600; httponly

  • cookie属性：

    • key/value：存储的键值对，如token=123456
    • domain：指定cookie生效的域名
    • path：指定cookie生效的路径
    • expire：过期时间，未设置则默认为会话结束
    • secure：为true时仅HTTPS请求携带
    • httponly：禁止JS访问，增强安全性

  • cookie传递机制：

    • 浏览器收到Set-Cookie后自动存储
    • 后续请求满足domain/path条件时自动附加
    • 示例请求头：cookie: token=123456; other=value…

  • 删除方法：

    • 服务器响应同名cookie但设置max-age=-1
    • 必须保持domain/path与原始cookie一致
    • 示例：set-cookie: token=; domain=yuanjin.tech; path=/; max-age=-1

  • 客户端设置：

    • 通过JS设置：document.cookie=”key=value; path=/; domain=xxx; max-age=3600”
    • 与服务器设置的区别：
      • 无httponly选项
      • path默认当前页面路径
      • domain默认当前域名

  • 实现步骤：

    • 创建登录接口接收账号密码
    • 验证成功后设置响应头：res.header(“set-cookie”, token=${result.id}; path=/; domain=localhost; max-age=3600)
    • 测试接口返回Set-Cookie头
    • 验证后续请求自动携带cookie

  • 注意事项：

    • 同域名下可能存在多个同名cookie（不同path）
    • 修改/删除时必须指定完整domain/path
    • 敏感信息应设置httponly和secure
6. 页面表单提交示例
1）传统表单提交实现
  • 表单结构：
  • 提交方式：使用传统form表单提交而非Ajax，提交地址为/api/admin/login，请求方法为POST。
2）登录请求与响应
  • 请求过程：
    • 输入测试账号abc和密码123123提交
    • 服务器返回JSON响应：
  • Cookie设置：
    • 响应头包含Set-Cookie: token=6; path=/; domain=localhost; max-age=3600
    • 浏览器自动保存该Cookie，有效期为3600秒（1小时）
3）Cookie的自动携带机制
  • 自动携带规则：
    • 当请求的域名、路径与Cookie设置匹配时，浏览器会自动在请求头中添加Cookie: token=6
    • 示例：请求/api/student时会自动携带该Cookie
  • 验证方法：
    • 在开发者工具的Network面板中查看请求头
    • 刷新页面时同样会携带Cookie
4）与传统Ajax方式的对比
  • 页面刷新：传统表单提交会导致页面刷新，而Ajax可以实现局部更新
  • 监控难度：页面刷新会使Network面板记录清空，需勾选”Preserve log”保持记录
  • Cookie机制：两种方式下Cookie的携带机制完全相同
7. 客户端设置cookie

  • 语法格式：通过document.cookie=”键=值; path=?; domain=?; expire=?; max-age=?; Secure”设置，与服务器端设置格式相同但存在差异

  • HTTPOnly限制：客户端设置时无法添加HTTPOnly属性，因为该属性本就是为限制客户端访问而设计

  • 路径默认值：未指定path时默认使用当前网页路径（不同于服务端使用请求路径）

  • 域名默认值：未指定domain时默认使用当前网页域名（原理同path）

  • 删除机制：通过设置max-age=-1或expire为过去时间实现删除，实际删除发生在下次请求时浏览器验证过期状态后

  • 删除验证：已过期的cookie仍会短暂显示，但在发送请求时浏览器会检查并清除过期cookie

  • 获取限制：通过document.cookie仅能获取键值对，无法读取path/domain等元信息

  • 特性对比：

    • 服务端设置的HTTPOnly cookie无法通过JS获取（如登录成功的token）
    • 客户端尝试设置HTTPOnly属性会被忽略（如cookie=”a=abc; httponly=true”无效）

  • 登录场景应用：

    • 浏览器发送账号密码到服务端
    • 服务端验证通过后通过响应头设置cookie（含HTTPOnly保护）
    • 后续请求自动携带cookie完成身份验证
8. 登录流程总结
1）Cookie原理及登录流程
  • 初始请求：
    • 浏览器发送POST请求到服务器，附带账号密码
    • 表单提交或Ajax请求均可实现，功能上没有区别
  • 服务器验证：
    • 验证账号密码正确性，错误则返回错误响应
    • 正确时在响应头设置Set-Cookie字段，包含登录认证信息（如用户ID）
  • 客户端存储：
    • 浏览器自动接收并存储Cookie，无需手动操作
    • 示例：document.cookie:”4297f44b13955235245b2497399d7a93”
2）后续请求处理机制
  • 自动携带机制：
    • 浏览器后续请求自动附加对应域名的Cookie
    • 示例：添加管理员请求会自动携带登录凭证
  • 服务器验证：
    • 先获取并验证Cookie信息有效性
    • 无效则拒绝操作，有效则执行正常业务流程
  • 认证信息设计：
    • 简单实现使用用户ID（存在安全隐患）
    • 推荐方案：JWT（JSON Web Token）令牌机制
3）Cookie管理特性
  • 多网站支持：
    • 客户端可存储来自不同网站的多个Cookie
    • 类似”卡包”存放多个”出入证”（每个Cookie对应一个凭证）
  • 自动管理功能：
    • 智能匹配：仅向对应网站发送关联Cookie（如肯德基凭证不发往麦当劳）
    • 过期处理：自动清理过期Cookie（通过max-age/expires属性）
  • 安全特性：
    • HttpOnly属性防止JS脚本访问敏感Cookie
    • Domain/Path限制Cookie的作用范围
4）服务器认证机制
  • 认证不认人原则：
    • 服务器通过令牌识别身份（类似脸盲者认证件不认人脸）
    • 完全依赖Cookie中的认证信息判断请求合法性
  • 令牌设计要点：
    • 需包含足够身份信息但不宜直接暴露敏感数据
    • 应设置合理有效期（示例代码中为3600秒）
    • 关键参数：path=/; domain=localhost; max-age=3600; httponly
