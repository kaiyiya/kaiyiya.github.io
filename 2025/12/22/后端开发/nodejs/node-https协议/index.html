一、对称加密

1. 通信安全的基本问题

  • 风险场景：以邓哥(客户端)向莫妮卡(服务器)发送”我爱你”为例，若传输过程被第三方(成哥)截获，可能被篡改为”我恨你”等恶意内容。
  • 明文缺陷：传统通信方式直接传输原始信息，第三方可轻松获取并修改内容，导致信息真实性无法保证。
2. 对称加密原理

  • 核心机制：使用单一密钥(
    keykeykey
    )进行加密和解密，数学表达式为：* 加密：
    加密信息=算法(明文+key)加密信息 = 算法(明文 + key)加密信息=算法(明文+key**)**
    • 解密：
      明文=算法(加密信息+key)明文 = 算法(加密信息 + key)明文=算法(加密信息+key**)**
  • 密钥特性：密钥是任意字符串，通过特定数学算法实现信息转换。
  • 常见算法：DES、3DES、AES、Blowfish等，算法差异仅在于计算步骤不同。
3. 对称加密应用示例

  • 实施步骤：
    • 服务器(莫妮卡)生成密钥
      key1key1key1
      并保存
    • 通信前先将
      key1key1key1
      传输给客户端(邓哥)
    • 后续通信使用
      key1key1key1
      加密/解密所有信息
  • 安全优势：
    • 加密后信息无法被第三方解读
    • 篡改会导致解密失败（产生错误而非错误结果）
  • 首次交换漏洞：若密钥
    key1key1key1
    在首次传输时被截获，攻击者同样能解密和篡改后续通信内容。
4. 对称加密的局限性

  • 根本缺陷：密钥分发过程存在安全风险，无法防范首次通信时的中间人攻击。
  • 典型攻击：攻击者获取初始密钥后：
    • 可解密所有通信内容
    • 能篡改信息后重新加密
    • 接收方无法识别信息被篡改
  • 错误处理：系统能识别解密失败的错误，但无法识别被成功解密的伪造信息。

二、非对称加密

  • 密钥对生成: 产生一对密钥，包括公钥和私钥两个部分，公钥用于加密，私钥用于解密
  • 密钥特性:
    • 公钥可公开分发，任何人都能获取
    • 私钥必须严格保密，泄露会导致严重安全隐患
  • 加密解密过程:
    • 加密：原始信息 + 公钥 → 加密信息
    • 解密：加密信息 + 私钥 → 原始信息
  • 算法特点: 具有数学上的非对称性，用公钥加密的信息不能用公钥解密
  • 常见算法: RSA、Elgamal、Rabin、D-H、ECC等，其中RSA应用最广泛
1. 通信安全应用

  • 基本通信流程:
    • 服务器生成密钥对（公钥k1，私钥k2）
    • 服务器将公钥k1发送给客户端
    • 客户端用k1加密信息（如”我爱你”）
    • 服务器用k2解密获取原始信息
  • 安全性分析:
    • 第三方无法解密：没有私钥k2
    • 但存在单向通信问题：服务器回复时客户端无法解密
2. 混合加密方案

  • 解决方案:
    • 客户端生成对称密钥k2
    • 用服务器公钥k1加密k2后传输
    • 服务器用私钥k2解密获取k2
    • 后续通信使用k2进行对称加密
  • 安全优势:
    • 密钥k2传输过程安全（公钥加密）
    • 后续通信效率高（对称加密）
    • 只有通信双方知道k2
3. 中间人攻击风险

  • 攻击原理:
    • 攻击者拦截服务器公钥k1
    • 伪造公钥k3发送给客户端
    • 客户端用k3加密的k2可被攻击者解密
    • 攻击者再用真实k1加密k2转发给服务器
  • 安全漏洞:
    • 可窃听：获取通信内容
    • 可篡改：修改传输信息
    • 根源：首次公钥传输无法验证真实性

三、CA机构

  • 定义: 证书颁发机构(Certificate Authority)是第三方权威机构，负责颁发数字证书。每个国家都有自己的CA机构。
  • 权威性: 浏览器和操作系统在安装时会记录全球已知的权威CA机构信息，确保其公信力。
1. 证书颁发流程

  • 申请条件:
    • 必须拥有有效域名(如www.monica.com)
    • 需要生成服务器密钥对(公钥key1和私钥KEY)
    • 需要支付费用(部分机构提供1年免费证书)
  • 申请材料:
    • 服务器地址
    • 服务器公钥
    • 申请费用
  • CA密钥对:
    • 公钥: 公开给全世界，包括浏览器和操作系统
    • 私钥: 绝对保密，用于加密证书信息
  • 安全机制: 攻击者即使攻破CA服务器也无法获取私钥，确保系统安全
  • 证书组成(DC):
    • 服务器地址(明文存储)
    • 证书颁发机构信息
    • 两个加密信息:
      • 服务器公钥(用CA私钥加密)
      • 证书签名(用CA私钥加密)
  • 防篡改机制:
    • 任何终端都可以用CA公钥解密证书内容
    • 但无法重新加密伪造，因为无法获取CA私钥
    • 确保信息只能读取不能修改
  • 签名验证:
    • 算法公开，使用服务器地址、CA公钥和服务器公钥生成
    • 终端可重新计算签名进行比对
    • 不一致则说明信息被篡改
  • 验证流程:
    • 获取证书签名
    • 使用公开信息重新计算
    • 比对计算结果
    • 不一致则拒绝证书

四、浏览器获取证书并验证

  • 证书获取：浏览器访问服务器时，服务器会返回其申请的证书（包含服务器公钥
    key1key1key1
    和证书签名），该证书只需申请一次即可重复使用。

  • 证书解析：浏览器内置多个证书颁发机构(CA)的公钥，利用CA公钥解密证书中加密部分，获取服务器公钥
    key1key1key1
    和证书签名。

  • 签名验证：通过CA公钥和服务器公钥
    key1key1key1
    按特定算法验证签名，若验证通过则说明证书未被篡改（包括CA公钥和服务器公钥的完整性）。

  • 防篡改特性：第三方无法篡改证书内容，因为证书关键部分（如公钥
    key1key1key1
    和签名）需用CA私钥加密，篡改后验证会失败并触发浏览器警告。

五、http协议和https协议的对比

1. http与https的基础

  • 协议层级：http直接建立在TCP/IP协议上，而https在TCP/IP之上增加了SSL/TLS加密层。
  • 加密流程：https在TCP三次握手后需完成证书验证、密钥交换等加密流程，之后才进行请求响应。
2. https的加密传输特性
  • 密钥交换：浏览器用服务器公钥
    key1key1key1
    加密对称密钥key2key2key2
    传输，服务器用私钥解密后，后续通信均使用key2key2key2
    加密。
  • 中间人防护：第三方无法获取
    key2key2key2
    （需私钥解密），也无法篡改加密数据（篡改会导致解密失败）。
3. 浏览器对https资源的处理
  • 混合内容策略：https网页中所有资源（如图片/CSS/JS）应使用https加载，否则浏览器会阻止或警告。
4. https无法解决的广告问题
  • 广告来源：
    • 浏览器/系统自身广告（https无法阻止）
    • 恶意软件注入（需杀毒解决）
    • 传输中被注入脚本（https可防护此类情况）
5. https的申请与证书

  • 服务器职责：需向CA申请证书，客户端只需在URL中使用https://前缀即可自动完成加密流程。
6. https的默认端口
  • 端口规范：默认使用443端口，若服务器监听其他端口需在URL中显式指定（如https://example.com:8443）。

六、知识小结

知识点 核心内容 考试重点/易混淆点 难度系数
HTTPS协议作用 保证数据传输过程中不被窃取和篡改，通过加密确保安全传输 明文传输 vs 加密传输的区别 ⭐⭐
对称加密 使用单一密钥进行加密和解密，如AES、DES算法 密钥分发安全问题 ⭐⭐⭐
非对称加密 使用公钥加密、私钥解密（如RSA算法），解决密钥分发问题 公钥与私钥的对应关系 ⭐⭐⭐⭐
CA证书机构 颁发数字证书验证服务器身份，防止中间人攻击 证书签名验证原理 ⭐⭐⭐⭐
HTTPS完整流程 1. 证书验证 → 2. 生成对称密钥 → 3. 加密通信 证书篡改检测机制 ⭐⭐⭐⭐⭐
HTTP与HTTPS对比 HTTPS=HTTP+SSL/TLS加密层，默认端口443 协议栈差异（TCP/IP vs SSL/TCP/IP） ⭐⭐
实际应用风险 混合内容（HTTP资源）导致浏览器警告 全站HTTPS必要性 ⭐⭐
