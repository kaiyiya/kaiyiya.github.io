一、Promise相关面试题

1. 面试题考点
  • 核心考察点: Promise是ES6面试题非常密集的考点，主要围绕核心原理展开
  • 常见题型:
    • 填空题/选择题/问答题：考察Promise的基本概念
    • 代码输出题：通过代码片段考察状态转换等概念理解
    • 状态转换题：考察Promise状态的不可变性
1）Promise的基本概念
  • 状态阶段
    • 未决阶段(unsettled): Promise创建后的初始状态，也称为挂起状态(pending)
    • 已决阶段(settled):
      • 完成状态(fulfilled): 通过resolve(data)触发，执行onFulfilled回调
      • 失败状态(rejected): 通过reject(reason)触发，执行onRejected回调
  • 重要特性
    • 状态不可变性: 一旦状态确定(resolve或reject被调用)，状态和数据都不能再改变
      • 示例：即使多次调用resolve/reject，只有第一次有效
    • 执行顺序: Promise构造函数中的同步代码会立即执行
  • 链式调用规则
    • 基本规则:
      • then方法必定返回一个新的Promise
      • 新Promise的状态取决于后续处理：
        • 无相关处理：与前任务状态一致
        • 有未执行处理：新任务挂起(pending)
        • 处理执行完成：
          • 无错：新任务完成(fulfilled)，数据为返回值
          • 有错：新任务失败(rejected)，数据为异常对象
          • 返回Promise对象：新任务状态与该对象一致
  • 静态方法
    • Promise.resolve(data): 直接返回一个完成状态的Promise
    • Promise.reject(reason): 直接返回一个拒绝状态的Promise
2）典型例题分析
  • 状态转换题
    • 解析要点:
      • Promise状态一旦确定不可更改
      • 多次调用resolve/reject只有第一次有效
      • 后续调用不会改变状态或数据
  • 执行顺序题
    • 解析要点:
      • Promise构造函数中的同步代码立即执行
      • then回调属于微任务，在当前宏任务结束后执行
      • setTimeout属于宏任务，执行顺序晚于微任务
2. 链式调用规则

  • 必然返回新Promise：.then()方法必定返回新的Promise对象，后续处理本身就是一个新任务
  • 状态传递机制：
    • 无后续处理时：新任务状态与前任务一致，数据继承前任务
    • 有未执行处理时：新任务处于挂起状态
    • 已执行处理时：
      • 成功情况：后续处理无错则新任务状态为完成，数据为处理返回值
      • 失败情况：后续处理有错则新任务状态为失败，数据为异常对象
  • 任务对象传递：若后续执行返回的是Promise对象，新任务的状态和数据与该返回对象保持一致
3. Promise的静态方法

  • 基础方法：
    • Promise.resolve(data)：直接返回完成状态的Promise，携带指定数据
    • Promise.reject(reason)：直接返回拒绝状态的Promise，携带拒绝原因
  • 批量处理方法：
    • Promise.all(任务数组)：
      • 全部成功则成功，任一失败则立即失败
      • 返回结果数组顺序与输入一致
    • Promise.any(任务数组)：
      • 任一成功即成功，全部失败才失败
      • 返回第一个成功的值
    • Promise.allSettled(任务数组)：
      • 等待所有任务最终状态（无论成功失败）
      • 永不失败，返回结果描述数组
    • Promise.race(任务数组)：
      • 采用第一个已决任务的状态和结果
      • 可用于超时控制场景
4. async和await

  • async函数特性：
    • 必定返回Promise对象
    • 普通返回值会自动包装为resolve状态
    • 返回Promise则保持原状态传递
  • await关键点：
    • 仅能在async函数中使用
    • 等待Promise决议：await 1 等价于 await Promise.resolve(1)
    • 错误处理需配合try-catch语法

5. 事件循环

1）任务队列分类
  • 宏任务队列(macro task):
    • 包含setTimeout和setInterval的回调函数
    • requestAnimationFrame的回调函数
    • 各种事件处理函数（如点击事件、鼠标移入事件等）
  • 微任务队列(micro task):
    • Promise的then回调函数
    • async/await本质也是Promise的语法糖，属于微任务
2）任务执行机制

  • 执行顺序:
    • 先执行同步代码
    • 然后检查并执行所有微任务
    • 最后执行一个宏任务
    • 循环往复
  • 特殊规则:
    • 计时器回调无论setTimeout还是setInterval都进入宏任务队列
    • Promise.then()回调是唯一进入微任务队列的情况
    • 事件处理函数默认进入宏任务队列
3）例题：Promise与事件循环
  • 题目解析
    • 同步代码执行顺序：1 → 2 → 4
    • 微任务执行顺序：3
    • 最终输出顺序：1 → 2 → 3 → 4
    • 易错点：容易忽略resolve()后的同步代码会继续执行
4）关键记忆点
  • 重要区别:
    • 宏任务：浏览器环境提供的API回调
    • 微任务：ES6新增的Promise相关回调
  • 执行优先级:
    • 微任务队列优先级高于宏任务队列
    • 同一批次微任务会全部执行完毕才执行宏任务
  • 常见考点:
    • Promise与setTimeout混合执行的顺序判断
    • async/await在事件循环中的表现
    • 事件处理与Promise的优先级比较

1）例题:promise输出结果
  • Promise执行机制解析
    • 执行栈与队列关系：Promise构造函数中的回调函数会立即执行，不进入事件队列；而.then()的回调属于微任务(micro task)，会进入微任务队列等待执行。
    • 状态变更时机：resolve()调用会立即改变Promise状态为fulfilled，但不会中断后续代码执行。如示例中resolve()后仍会执行console.log(2)。
    • 微任务执行时机：即使Promise状态已变为fulfilled，.then()回调仍需等待当前执行栈清空（全局代码执行完毕）才会从微任务队列取出执行。
  • 例题解析方法
    • 解题工具：
      • 三列表法：建议用执行栈、宏队列、微队列三个列表记录执行过程
      • 状态记录：需额外记录Promise对象的状态变化（pending→fulfilled/rejected）和返回值
    • 关键步骤：
      • 同步代码按顺序执行并记录输出
      • 遇到.then()时将回调加入微队列
      • 同步代码执行完毕后检查微队列
      • 微任务执行可能产生新的微任务（链式调用时需注意）

5）例题：Promise输出示例

  • 初始状态分析
    • Promise状态初始化：
      • const promise1 = new Promise()创建时处于pending状态
      • setTimeout(()=>{resolve()},1000)表示1秒后将执行resolve
      • promise2 = promise1.catch()创建时同样处于pending状态
    • 执行顺序：
      • 先同步执行console.log(‘promise1’, promise1)输出pending状态
      • 同步执行console.log(‘promise2’, promise2)输出pending状态
      • 1秒后执行resolve改变promise1状态
      • 2秒后执行最终状态输出
  • 链式调用规则
    • 核心规则：
      • 规则1：.then/.catch必定返回新Promise，形成任务链
      • 规则2：新任务状态取决于后续处理：
        • 无相关处理：新任务状态与前任务一致
        • 有未执行处理：新任务挂起(pending)
        • 已执行处理：根据处理结果确定新状态
      • 规则3：后续处理无错→新任务fulfilled，返回值为数据
        • 有错→新任务rejected
        • 返回Promise→等待该Promise结果
  • 题目解析(resolve情况)
    • 执行过程：
      • 1秒后执行resolve()→promise1变为fulfilled(undefined)
      • 检查promise1.catch()：
        • promise1已fulfilled，但.catch()是rejected回调→不执行
        • 根据规则2无相关处理→promise2状态与promise1一致
      • 2秒后输出：
        • promise1: fulfilled undefined
        • promise2: fulfilled undefined
    • 关键点：
      • 回调函数是否执行取决于前Promise的最终状态
      • 无对应状态回调时，链式Promise状态会”穿透”
  • 题目变体(reject情况)
    • 变化点：
      • 将resolve()改为reject()→promise1变为rejected(undefined)
    • 执行过程：
      • 1秒后执行reject()→promise1变为rejected
      • 检查promise1.catch()：
        • promise1已rejected→执行catch回调()=>{return 2}
        • 回调正常执行→promise2变为fulfilled(2)
      • 2秒后输出：
        • promise1: rejected undefined
        • promise2: fulfilled 2
    • 关键区别：
      • rejected状态会触发.catch()回调执行
      • 回调返回值决定链式Promise的最终状态
  • 静态方法补充
    • 常用静态方法：
      • Promise.resolve(data)：直接返回fulfilled状态的Promise
      • Promise.reject(reason)：直接返回rejected状态的Promise
      • 适用于快速创建已确定状态的Promise对象
  • 事件循环机制
    • 执行顺序：
      • 同步代码→微任务(Promise)→宏任务(setTimeout)
      • 题目中：
        • 第一轮：同步输出pending状态
        • 第二轮：1秒后resolve/reject→检查微任务
        • 第三轮：2秒后输出最终状态
    • 记忆口诀：
      • “同微宏”三字诀：同步→微任务→宏任务
      • Promise回调属于微任务，setTimeout属于宏任务
6）例题:await用法
  • 基础概念解析
    • 语法糖本质：async/await本质上是Promise的语法糖，await 1等价于Promise.resolve(1).then(n => {…})
    • 执行机制：遇到await表达式时，函数会暂停执行但不阻塞线程，后续代码会被放入微任务队列
    • 状态变化：async函数返回的Promise在内部所有await完成前保持pending状态
  • 例题解析1
    • 题目解析
      • 执行顺序：
        • 立即执行m()函数，同步输出0
        • 遇到await 1将后续代码放入微任务队列
        • 继续执行外层同步代码输出2
        • 最后执行微任务输出1
      • 关键点：await不会阻塞后续同步代码执行
      • 答案：输出顺序为0 → 2 → 1
7）例题:await用法
  • 嵌套执行分析
    • 多层await：当async函数被另一个async函数调用时，会形成Promise链
    • 状态传递：内层Promise的状态决定外层await的继续时机
    • 微任务队列：每个await都会产生新的微任务，按完成顺序依次执行
  • 例题解析2
    • 题目解析
      • 执行流程：
        • 立即执行IIFE，调用m()输出0
        • await 1将后续代码放入微任务队列1
        • 继续执行外层同步代码输出3
        • 执行微任务队列1输出1
        • 内层Promise解决后触发外层await继续，输出2
      • 状态观察：在console.log(3)时，m()返回的Promise仍为pending状态
      • 答案：输出顺序为0 → 3 → 1 → 2
  • 核心要点总结
    • 永不阻塞原则：JavaScript永远不会真正阻塞执行，await只是语法糖
    • 微任务优先级：await产生的微任务优先于setTimeout等宏任务执行
    • 转换规律：任何async函数都可转换为等效的Promise链式调用形式
    • 状态判断：async函数返回的Promise在最后一个await完成前保持pending
8）例题：await用法

  • async函数返回值：async函数必定返回Promise对象，返回值会被自动包装成Promise.resolve()
  • await执行机制：
    • 阻塞后续代码：await后的代码会暂停执行，直到Promise状态变为fulfilled
    • 微任务队列：await后面的代码会被放入微任务队列
    • 值获取：await表达式会返回Promise的解决值
  • 无await情况：
    • 直接获取Promise对象本身
    • 不会阻塞后续代码执行
    • 函数会立即返回Promise

    • 链式调用本质：方法调用返回的是最后一个Promise对象，而非中间对象。如b = new Promise().then().then()中，b指向最后一个then返回的Promise。

      • pending：初始状态，可通过resolve转为fulfilled或通过reject转为rejected
      • fulfilled：不可逆状态，会触发then注册的成功回调
      • rejected：不可逆状态，会触发catch注册的失败回调
    • 微队列机制：
      • 每个状态变化会将对应回调加入微队列
      • 当前宏任务执行完毕后按序执行微队列
    • 死锁场景：
      • 当Promise等待自身完成时（如await a且a处于pending）
      • 类似”没有口罩不能出门，不出门买不到口罩”的逻辑矛盾
  • 异步函数执行特点
    • async函数特性：
      • 自动将返回值包装为Promise
      • await会暂停当前函数执行，但不阻塞主线程
    • 执行顺序陷阱：
      • 变量赋值在表达式完全计算后才进行
      • await表达式会分解为多个微任务
    • 调试技巧：
      • 用console.log确认Promise当前状态
      • 通过断点观察微任务队列变化
      • 标记Promise对象便于跟踪（如p1-p5）
12）例题:async/await用法
  • 异步执行机制
    • 执行顺序原则：
      • JS代码永远不会卡住，不会等待异步操作完成才继续执行
      • 遇到异步任务会先将其放入相应队列（宏队列/微队列），继续执行后续同步代码
      • 同步代码执行完毕后，再从队列中取出异步任务执行
    • 队列优先级：
      • 微队列（Promise.then、async/await）优先于宏队列（setTimeout）
      • 同一队列中按先进先出顺序执行
  • async/await本质
    • 语法糖特性：
      • await只是语法糖，表面看起来代码在”等待”，实际是Promise的封装
      • 底层实现相当于在.then()中执行后续代码
      • 示例：await async2() 等价于 async2().then(() => { /* 后续代码 */ })
    • 执行流程：
      • 遇到await会暂停当前async函数的执行（但不会阻塞主线程）
      • 将await后面的表达式求值并包装为Promise
      • 将await之后的代码作为微任务放入队列
  • 例题解析
    • 题目解析：
      • 同步代码立即执行：先输出’script start’、’promise1’等
      • setTimeout回调进入宏队列
      • async函数中的await会将其后代码放入微队列
      • 执行顺序：同步代码 > 微队列 > 宏队列
      • 关键点：即使Promise已resolve，相关回调仍需进入队列等待执行
    • 易错点：
      • 误认为await会阻塞后续所有代码执行
      • 混淆微任务和宏任务的执行时机
      • 忽略Promise立即resolve时.then回调仍需进入队列的特性
  • Promise状态变化
    • 三种状态：
      • pending（挂起状态）：初始状态
      • fulfilled（完成状态）：调用resolve(data)后进入
      • rejected（失败状态）：调用reject(reason)后进入
    • 状态特性：
      • 状态一旦改变就不可逆（从pending到fulfilled或rejected）
      • 可以通过.then()添加状态改变时的回调
      • .catch()专门处理rejected状态
  • 事件循环机制
    • 宏任务(macro task)：
      • setTimeout/setInterval回调
      • I/O操作
      • UI渲染
      • 事件处理函数
    • 微任务(micro task)：
      • Promise.then/catch/finally回调
      • async/await转化后的回调
      • MutationObserver
    • 执行规则：
      • 每执行一个宏任务后，会清空整个微任务队列
      • 微任务执行期间产生的新微任务会加入当前队列继续执行
      • 宏任务执行完毕后会进行UI渲染
