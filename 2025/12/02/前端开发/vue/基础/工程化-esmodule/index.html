一、模块化手册

1. 模块化标准对比

  • CommonJS特性:
    • 标准类型: 社区规范（民间自发制定）
    • 支持环境: 仅Node.js环境支持
    • 依赖类型: 动态依赖（运行时加载）
    • 导出语法: 通过module.exports导出单个值
    • 导入语法: 使用require(“模块路径”)获取导出值
  • 开发优势:
    • 将复杂功能拆分为独立模块，提高代码复用性
    • 保持每个JS文件代码简洁，便于大型工程维护
    • 避免全局变量污染问题
2. ES Module详解

  • 核心特性:
    • 标准类型: 官方标准（ES6规范组成部分）
    • 支持环境: Node.js和浏览器均支持
    • 依赖类型: 支持静态依赖（编译时分析）和动态依赖
  • 导出方式:
    • 具名导出:
      • 可导出多个值：export const a=1;、export function b(){}
      • 支持导出重命名：export { k as temp }
    • 默认导出:
      • 每个模块只能有一个默认导出
    • 混合导出:
      • 两种导出方式可共存，最终合并为单个导出对象
3. 模块化发展历史
  • 发展背景:
    • CommonJS最初由Node社区创建
    • 前端开发者推动官方标准化（2015年随ES6发布）
  • 历史阶段:
    • 早期民间标准：AMD/CMD等（已逐渐淘汰）
    • 现行标准：CommonJS（Node主流）、ES Module（官方标准）
  • 学习建议:
    • 虽然Node开发较少，但CommonJS必须掌握
    • ES Module语法更复杂但功能更强大
    • 可通过文档查询辅助记忆，无需死记硬背
4. 语法对比实践
  • CommonJS优势:
    • 语法简单直观，初学者友好
    • 导出/导入仅需掌握两个核心语法
  • ES Module特点:
    • 提供更灵活的导出方式（多值导出/默认导出）
    • 支持静态分析优化
    • 浏览器原生支持特性
  • 注意事项:
    • 两种标准不能混用（需统一使用同种规范）
    • 默认导出本质仍是对象属性default的简写
5. 官方标准
1）工程练习

  • CommonJS标准：
    • 类型：社区规范
    • 支持环境：仅Node.js
    • 依赖类型：动态依赖
    • 导出方式：module.exports = 导出的值
    • 导入方式：require(“模块路径”)返回模块导出的值
  • ES Module标准：
    • 类型：官方标准（ECMAScript规范）
    • 支持环境：Node.js和现代浏览器
    • 依赖类型：支持静态和动态依赖
    • 兼容性：2015年后浏览器基本支持，之前版本不兼容
2）浏览器环境
  • 模块化实践
    • 环境搭建：
      • 需创建HTML页面作为入口（浏览器无法直接运行JS文件）
      • 传统JS引用方式：<script src="path.js"></script>
    • ES Module引用：
      • 需添加type属性：<script src="path.js" type="module"></script>
      • 特性：浏览器会将JS文件作为模块处理（启动模块）
  • 模块作用域验证
    • 模块特性验证：
      • 模块内var a = 1不会污染全局作用域
      • 控制台访问a报错：Uncaught ReferenceError
      • window.a结果为undefined
    • 模块化本质：
      • 所有变量默认模块作用域
      • 暴露数据必须显式导出
      • 使用外部模块必须显式导入
  • 实践练习
    • 练习要点：
      • 创建HTML文件+JS模块文件
      • 通过type=”module”引用JS
      • 验证模块作用域特性

3）ES Module的导出

  • 分屏演示

    • 模块关系：演示index.js和math.js两个模块的交互，math模块需导出sum求和与isOdd判断奇偶两个函数
    • CommonJS对比：CommonJS通过module.exports赋值导出，而ES Module采用不同语法规则

  • ES Module的导出

    • 核心机制：模块最终合并为一个对象导出，包含default默认导出和具名导出两种属性
    • 具名导出：
      • 直接导出：export const a=1 或 export function b(){}，变量名即导出名
      • 后置导出：先定义const d=2，再通过export {d}导出
      • 重命名导出：export {k as temp} 将变量k以temp名称导出
    • 默认导出：
      • 单次限制：每个模块只能有一个export default
      • 语法变体：export default 3 或 export default function(){}
      • 等效写法：const e=4; export {e as default}

4）静态依赖和动态依赖
  • 静态导入语法：
    • import c from “模块路径”：导入默认导出(default)到变量c
    • import c, {a,b} from “模块路径”：同时导入默认导出和具名导出
    • import * as obj from “模块路径”：将整个模块对象导入到变量obj
    • import {a as temp1, b as temp2} from “模块路径”：导入具名导出并重命名
    • import {default as a} from “模块路径”：导入默认导出必须使用别名
  • 动态导入：
    • import(“模块路径”)：返回Promise，模块对象在resolve时获取
    • 特点：可在代码块中执行，不要求顶级作用域
  • 静态依赖特点
    • 代码位置限制：必须写在模块顶级作用域，不能放入判断/循环/函数等代码块中
    • 原因：需要在代码运行前分析出所有依赖关系
    • 分析时机：代码未运行时就要确定模块间的依赖关系
  • 动态依赖特点
    • CommonJS特性：
      • 使用require()函数动态导入
      • 依赖关系在代码运行时才能确定
      • 可放入条件判断、循环等代码块中执行
    • 与ESM区别：
      • 静态依赖：编译时分析依赖
      • 动态依赖：运行时确定依赖
  • 导出语法
    • 具名导出：
      • export const a = 1
      • export function b(){}
      • 可导出多个，需使用{}导入
    • 默认导出：
      • export default {…}
      • 只能导出一个，可直接导入
    • 混合导出：可同时使用两种导出方式，最终合并为一个对象
  • 注意事项
    • 导出限制：导出代码必须为顶级代码，不可放入代码块
    • 特殊导入：
      • import “模块路径”：仅执行模块，不导入任何内容
      • import()函数：ES2020新增的动态导入方式
    • 默认导出处理：
      • default是关键字，不能直接作为变量名
      • 必须使用import {default as 别名}语法
5）动态导入

  • 动态导入的概念

    • 静态导入限制：ES Module的静态导入必须放在代码顶端，不能放入代码块中，否则会报语法错误。
    • 动态导入优势：ES Module标准更新后支持动态导入，允许在代码块中按需加载模块，解决了静态导入的位置限制问题。

  • 动态导入的Promise返回

    • 异步特性：动态导入返回一个Promise对象，因为需要异步加载模块文件。
    • 模块对象获取：Promise完成时返回的是整个模块对象，包含所有导出内容。
    • 与Node.js区别：浏览器环境下必须异步加载，而Node.js在服务器环境可以同步加载。

三、模块划分知识总结

1. 模块化核心目的
  • 核心价值：
    • 分解大型代码文件
    • 避免全局变量污染
    • 提高代码复用性
  • 划分原则：
    • 功能相似/相同的代码集中
    • 差异功能分离到不同模块
    • 过大模块可人为分解
2. 模块化方案对比
  • CommonJS：
    • 民间方案
    • 语法简单（require/exports）
    • 主要用于Node.js环境
  • ES Module：
    • 官方标准方案
    • 使用import/export语法
    • 现代前端开发主流选择
  • 发展趋势：ES Module将成为长期使用的标准方案，必须重点掌握
