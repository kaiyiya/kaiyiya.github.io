一、模块化学习

1. 问题引入
1）js文件代码行数掌控问题
  • 掌控范围：普通开发者对单个代码文件的掌控能力通常在500行以内，学生阶段可能在200-300行以内
  • 代码量过多的影响：
    • 阅读性差：需要频繁使用滚动条定位代码位置
    • 功能混杂：多个功能混杂在同一个文件中难以区分和维护
    • 维护困难：代码量越大越难进行修改和调试
2）代码文件数量多的问题及影响
  • 解决方案：将大型项目拆分为多个小文件
  • 拆分优势：
    • 每个文件专注单一功能
    • 便于团队协作开发
    • 降低单个文件的复杂度
  • 拆分后新问题：
    • 全局变量污染
    • 依赖关系混乱
3）全局变量污染问题
  • 定义：多个JS文件共享同一全局命名空间导致的变量/函数命名冲突
  • 示例：
    • 文件A定义全局函数getRandom(min, max)
    • 文件B也定义同名全局函数getRandom
    • 后加载的文件会覆盖先加载的同名函数
  • 影响：
    • 开发者不敢随意定义全局变量
    • 团队协作时难以管理全局命名空间
    • 第三方库可能引入不可预见的命名冲突
4）依赖混乱问题
  • 表现：
    • 文件之间存在复杂的依赖关系（如B依赖A，C依赖B等）
    • 需要手动维护正确的加载顺序
  • 实际困难：
    • 大型项目可能有上百个JS文件
    • 依赖链可能非常深（多层嵌套依赖）
    • 新增文件时难以确定正确的插入位置
  • 后果：
    • 导致前端程序复杂度受限
    • 难以开发大型复杂应用
5）模块化的出现与意义
  • 核心价值：
    • 解决全局污染：每个模块有独立作用域
    • 解决依赖混乱：提供明确的依赖声明机制
  • 开发优势：
    • 支持代码细分为多个小文件
    • 每个文件专注单一功能
    • 便于团队分工协作
    • 使复杂系统开发成为可能
  • 历史意义：
    • 突破了前端开发的规模限制
    • 使大型复杂应用（如微博、知乎等）的前端开发成为可能
2. 前端模块化的标准
1）模块化的标准

  • 前端模块化标准

    • CommonJS

      • 定义: 简称CMJ，是一种社区制定的模块化标准
      • 产生背景:
        • 由于ES官方早期未推出模块化标准
        • 前端社区为解决实际问题自发形成
      • 现状:
        • 目前仍被约10%的项目使用
        • 面试中考察频率较高
        • 是Node.js默认采用的模块系统
      • 特点:
        • 出现时间早于ES Module
        • 采用同步加载方式
        • 使用require()导入，module.exports导出

    • ES Module

      • 定义: 简称ESM，是ECMAScript官方模块标准

      • 特点:

        • 采用import/export语法
        • 支持静态分析和tree shaking
        • 使用异步加载机制

      • 市场占比:

        • 约占当前项目使用量的90%
        • 是现代前端开发的主流选择

      • 与CMJ区别:

        • 官方标准vs社区标准
        • 静态分析vs动态加载
        • 异步vs同步
        • 浏览器原生支持vs需要打包工具处理

      • 学习建议:

        • 两种标准都需要掌握
        • CommonJS常见于遗留系统和Node.js环境
        • ESM是未来发展方向
        • 实际项目中可能混合使用
2）CommonJS如何实现模块化

  • node环境

    • node环境支持CommonJS

      • 环境限制：CommonJS(CMJ)是社区规范，仅能在node环境中使用，浏览器环境不支持
      • 学习必要性：虽然不开发node应用，但后续构建工具需要使用CommonJS，必须掌握基础用法
      • 与ESM区别：ES Module(ESM)是官方标准，浏览器和node环境都支持，但CommonJS出现更早

    • 安装node环境

      • 安装必要性：CommonJS模块化必须在node环境运行，需先确保node安装成功
      • 版本选择：推荐下载长期支持版(LTS)，如14.16.1版本，稳定性更高
      • 下载地址：官网https://nodejs.org/zh-cn/提供各平台安装包

    • 查看node版本

      • 验证命令：在终端输入node -v查看已安装版本
      • 成功标志：输出版本号如v14.16.1即表示安装成功
      • 异常处理：若无版本输出，需检查安装或重新下载安装包

    • node环境运行JS代码

      • 运行方式：通过node 文件名.js命令直接执行JS文件
      • 环境特点：无需浏览器参与，代码在node运行时环境中执行
      • 基础示例：console.log(‘Hello Node!’)可直接输出到终端

    • 使用code runner运行JS代码

      • 插件功能：Code Runner插件实质是在node环境执行代码
      • 操作方式：右键选择”Run Code”即可执行当前JS文件
      • 输出位置：执行结果会显示在VS Code的输出面板中

    • 使用终端运行JS代码

      • 标准方法：在文件所在目录打开终端，输入node 文件名.js

      • 路径要求：需确保终端当前目录与JS文件所在目录一致

      • VS Code集成：可直接使用VS Code内置终端执行node命令

      • 语言核心：ES语法部分在node和浏览器中完全一致

      • API差异：浏览器提供DOM/BOM等Web API，node提供文件系统等服务器API

      • 执行环境：node自带V8引擎，不需要浏览器即可解析执行JS代码

  • 应用案例

    • CommonJS模块系统

      • 模块基本概念

        • 入口模块：通过node命令直接运行的文件称为入口模块或启动文件，如index.js。在Node环境中只能运行一个入口模块。
        • 模块独立性：每个.js文件都是一个独立模块，具有特定功能单元（如math.js专门处理数学函数）。

      • CommonJS规范

        • 规范来源：CMJ是社区标准，目前仅Node环境完整实现
        • 模块隔离性：
          • 所有模块中的全局变量/函数不会污染其他模块
          • 测试示例：在global对象中查不到模块内定义的var a=1和function m(){}

      • 模块导出机制

        • 导出方式：使用module.exports赋值
          • 可导出任意类型（数字、字符串、对象、函数等）
          • 导出多个内容需使用对象包裹（如{isOdd, sum}）
        • 导出原理：模块运行完成后，module.exports的最终值即为导出内容

      • 模块导入机制

        • 导入语法：require(‘模块路径’)
        • 路径规则：
          • 必须以./或../开头
          • 可省略.js后缀（默认补全）
        • 执行流程：
          • 加载并执行目标模块文件
          • 返回该模块的module.exports对象
          • 通常将返回值赋给变量（建议变量名与模块名一致）

      • 模块依赖关系

        • 依赖链：入口模块可依赖其他模块，被依赖模块又可继续依赖更多模块
        • 优势：
          • 避免全局污染
          • 实现功能拆分
          • 支持多人协作开发
        • 典型应用：
          • math.js提供数学函数
          • index.js作为入口协调各模块

    • CommonJS模块化基础

      • 模块导出与导入

        • 导出语法：使用module.exports = { 函数名, 变量名 }将模块内容导出
        • 导入语法：通过const 变量名 = require(‘./模块路径’)导入其他模块
        • 路径规范：模块路径必须以./或../开头，表示相对路径

      • 模块缓存机制

        • 缓存原理：Node.js会缓存第一次运行的模块结果
        • 运行特点：
          • 首次require时执行模块代码并缓存导出结果
          • 后续require直接返回缓存结果，不再重复执行模块代码
        • 验证示例：多次require同一模块时，控制台只输出一次”math run”

      • 模块化优势

        • 解决痛点：
          • 避免全局变量污染
          • 管理复杂的依赖关系
        • 适用场景：当代码量达到成千上万行时，模块化能有效细分代码结构
        • 主要标准：
          • CommonJS：Node.js原生支持
          • ES Module：浏览器和Node.js新版支持

      • 练习要求

        • config模块：需导出包含wordDuration和text的对象
        • delay模块：需要导出延迟函数
        • 实现要点：
          • 每个模块文件保持独立功能
          • 通过require/export建立模块间联系
          • 体验模块缓存的实际效果

    • 例题:配置模块与延迟模块导出

      • 配置模块config.js

        • 模块功能：创建一个配置文件模块，用于存储打印文本和时间间隔配置

        • 实现步骤：

          • 新建config.js文件
          • 导出包含两个属性的对象：
            • wordDuration:300：打印每个字的时间间隔（毫秒）
            • text：要打印的完整文本内容（示例为《忆秦娥·娄山关》词句）

        • 模块特点：

          • 通过修改配置对象即可调整程序行为，提高可维护性
          • 独立配置模块使其他模块可以复用这些配置

        • 导出方式：使用CommonJS的module.exports导出配置对象
