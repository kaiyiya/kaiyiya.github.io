工程化-webpack

一、构建工具

1. 工程化

  • 本质目的: 工程化不是让开发变复杂，而是为了解决复杂应用中的问题。当应用简单时（如写一个HTML页面引用CSS/JS），问题被掩盖；当应用复杂时，这些问题就会暴露。
  • 适用场景: 开发简单功能（如轮播图、Hello World）不需要工程化，但复杂项目必须使用工程化方案。
1）构建工具

  • 核心定位: webpack是前端工程化的集大成者，用于搭建完整的前端工程体系。
  • 运行环境: 基于Node.js运行环境，主要功能是对项目资源进行打包处理。
  • 工作流程:
    • 入口分析: 以指定模块（如src/main.js）为起点
    • 依赖收集: 自动分析所有模块依赖关系
    • 资源处理: 对CSS、图片、音频等各类资源进行合并压缩
    • 输出结果: 生成优化后的最终打包文件
2）构建工具的作用

  • 问题解决者: 暴露并解决前端开发中长期被忽视的问题（如模块依赖、资源优化等）
  • 开发提效:
    • 提供开发服务器（Dev Server）
    • 支持文件缓存和热更新
    • 自动处理路径别名和文件后缀
  • 质量保障:
    • 生成source map便于调试
    • 自动处理CSS兼容性（厂商前缀）
    • 支持CSS Modules等现代方案
  • 学习建议:
    • 课程为简化理解会牺牲部分表述准确性
    • 完整掌握需要系统学习（约1个月周期）
    • 基础工程习惯：所有模块应放置在src目录
2. webpack介绍
1）webpack工程搭建体验

  • 工程结构说明

    • 工程特点：提供已搭建好的前端工程模板，使用VS Code打开后可见完整项目结构
    • 文件说明：工程中包含多种配置文件，这些文件的作用会在后续课程中逐步解释
    • 学习重点：重点在于学会使用已有工程，而非从零搭建工程

  • webpack基本概念

    • 运行环境：webpack运行在Node.js环境中，具有文件读写能力
    • 核心功能：将工程中大量文件进行压缩合并，形成少量优化后的文件
    • 打包过程：这个过程称为”打包”，是webpack的核心价值所在
    • 模块处理：能够处理JS、图片、音视频、CSS、HTML等各种资源文件
2）webpack打包体验

  • 打包流程演示

    • 第一步：安装依赖：通过npm i命令安装项目所需的所有依赖包
    • 第二步：编写模块：在src目录下创建多个模块（JS、图片、样式等）
    • 第三步：运行打包：执行npm run build命令启动webpack打包过程
    • 第四步：查看结果：打包生成的文件会输出到dist目录中

  • 工程目录结构

    • src目录：存放所有源代码的目录，包含JS、CSS、图片等各种资源
    • 入口文件：默认以src/main.js作为打包入口文件
    • 依赖关系：webpack会从入口文件开始分析所有模块的依赖关系

  • 打包结果分析

    • 输出目录：打包结果默认生成在dist目录中
    • 文件变化：原始多个文件会被合并为少量优化后的文件
    • 代码转换：模块化语句会被移除，转换为浏览器可直接运行的代码

  • webpack优势总结

    • 模块化支持：可以自由使用任意模块化标准，无需担心兼容性问题
    • 资源统一处理：将非JS代码（CSS、图片等）也视为模块进行管理
    • npm集成：支持直接使用npm安装的第三方模块
    • 单页应用：特别适合开发用户体验良好的单页应用(SPA)

  • 实际打包演示

    • 样式文件处理：通过导入Less文件演示CSS资源的打包过程
    • 图片资源处理：演示图片文件的导入和打包结果
    • 最终效果：打包后会生成可直接在浏览器中运行的HTML、CSS和JS文件

  • 开发工作流

    • 开发阶段：专注于在src目录中编写源代码，使用现代语法和模块化
    • 构建阶段：通过webpack将源代码转换为生产环境可用的优化版本
    • 部署阶段：只需部署dist目录中的内容到服务器即可
3）构建工具的好处

  • 模块化标准的使用

    • 标准兼容性：可以同时使用ES6模块化(import/export)和CommonJS模块化(require/module.exports)
    • 打包处理机制：webpack会将所有模块化语句在打包过程中移除，最终生成传统JS代码
    • 兼容性优势：由于打包后代码不包含任何模块化语法，因此无需考虑浏览器对模块化的支持情况
    • 推荐标准：虽然支持多种标准，但建议优先使用ES6模块化，因为这是官方标准

  • 非JS代码作为模块

    • 资源类型扩展：可以将CSS、LESS、图片等非JS文件视为模块导入
    • 开发优势：
      • 允许对样式文件进行细粒度划分（如按组件拆分CSS）
      • 打包时会自动合并多个CSS文件，避免因文件过多导致的请求次数增加
      • 示例：创建a.less文件并在JS中导入，打包后会合并到最终CSS中
    • 构建过程：webpack会处理这些非JS资源的依赖关系，最终生成优化后的打包结果

  • 前端使用npm

    • 使用方法：
      • 通过npm安装依赖（如jquery）
      • 使用import语句导入模块
      • 像使用普通模块一样使用npm包
    • 优势特点：
      • 不再需要手动下载JS文件并在HTML中引用
      • 所有第三方库都会被webpack视为依赖并合并到最终打包结果中
      • 示例：安装jquery后创建并添加h1元素到页面
    • 打包机制：webpack会将依赖的第三方库代码与项目代码合并，生成一个包含所有依赖的JS文件

  • 适合开发单页应用

    • 单页应用特点：
      • 整个应用只有一个HTML页面
      • 所有内容通过JS动态生成和更新
      • 页面切换时不刷新，仅更新DOM内容
    • 开发优势：
      • 与前端框架（如Vue、React）完美配合
      • 提供更好的用户体验（无页面刷新）
      • 示例：通过JS动态创建并添加h1元素到页面
    • 趋势说明：单页应用已成为现代前端开发的主流模式
3. 构建工具的使用
1）应用案例
  • 例题:构建工具的使用

    • Webpack打包流程

      • 导入模块: 导入JS模块、CSS、图片和第三方库（如jQuery）。
      • 使用第三方库: 使用jQuery创建并插入元素到DOM中。
      • 打包结果: Webpack会生成一个包含所有依赖的打包文件，并可以将其复制到任何地方运行。

    • 页面模板

      • 单页应用: 对于单页应用，只有一个空白的页面，所有内容都靠JS代码创建。
      • 自动生成页面: Webpack会自动生成一个页面，并在页面中自动加入对JS和CSS的引用。
      • 页面模板: 生成页面时，参考的是public/index.html，称之为页面模板。
      • 复制文件: Webpack会将public目录中的所有文件（除页面模板外）复制到打包结果中。

    • 开发阶段体验

      • 开发服务器: 在开发阶段，可以运行npm run serve命令获得更好的打包体验。
      • 内存打包: Webpack不会形成打包结果文件，而是把打包的内容放到内存中。
      • 自动刷新: 当源码发生变动时，Webpack会自动重新打包，并刷新页面以访问到最新的打包结果。

    • Public目录

      • 目录特点: public目录中的所有文件（除页面模板外）会被原封不动地复制到打包结果中。
      • 文件处理: 例如，图标文件（favicon.ico）和其他文件都会被复制。
      • 无需改动: 通常不需要改动public目录，它里面保留一个index.html即可。

    • 页面自动生成与引用

      • 自动生成页面: Webpack会自动生成一个页面，页面内容靠JS代码创建。
      • 自动引用: Webpack会在页面中自动加入对JS和CSS的引用。
      • 延迟执行: JS文件通过defer属性实现延迟执行，即等页面加载完后再运行JS。
2）开发服务器

  • 开发服务器简介

    • 作用：解决传统开发流程中”修改代码->手动打包->运行查看”的繁琐过程，提供更高效的开发体验
    • 特点：在开发阶段使用，通过npm run serve命令启动，实现代码修改后自动刷新页面

  • 源码目录与打包过程

    • 目录结构：源代码存放在src目录下，包含各种资源文件（JS、CSS、图片等）
    • 传统流程问题：每次修改都需要手动运行npm run build打包，再将dist目录作为根目录打开查看效果

  • npm run serve命令

    • 命令作用：启动开发服务器，提供更好的开发体验
    • 使用方式：在package.json中配置好serve命令后，直接在终端运行npm run serve
    • 访问方式：命令运行后会提示访问地址（通常是http://localhost:8080），按住ctrl键点击即可在浏览器打开

  • 开发服务器自动刷新

    • 实时更新：修改代码并保存后，浏览器页面会自动刷新显示最新效果
    • 优势：无需手动打包和刷新，提高开发效率
    • 停止服务：在终端按ctrl+c可停止开发服务器

  • 开发服务器原理

    • 核心机制：webpack启动一个本地开发服务器（dev server）
    • 监听端口：默认监听8080端口，通过该端口提供打包结果
    • 请求流程：浏览器访问localhost:8080 -> 请求开发服务器 -> 服务器响应打包结果

  • 开发服务器与内存打包

    • 内存打包：开发模式下webpack不会生成实际打包文件，而是将打包结果保存在内存中
    • 性能优势：内存操作比磁盘IO更快，提升开发效率
    • 资源获取：服务器直接从内存中读取打包结果响应给浏览器

  • 开发服务器请求响应流程

    • 完整流程：
      • 浏览器请求页面资源
      • 开发服务器从内存打包结果中取出对应资源
      • 服务器响应资源给浏览器
      • 页面加载后请求JS/CSS等附加资源
      • 服务器再次从内存响应这些资源

  • 开发服务器对开发的影响

    • 开发体验：大幅简化开发流程，实现”修改即见效果”
    • 使用要点：
      • 仅适用于开发阶段
      • 生产环境仍需使用npm run build生成实际打包文件
      • 打包结果需作为根目录部署

  • 作业：使用开发服务器

    • 实践内容：
      • 运行npm run serve启动开发服务器
      • 在浏览器中查看效果
      • 修改代码体验自动刷新
      • 使用ctrl+c停止服务
3）文件缓存
  • 哈希值作用：解决浏览器缓存导致代码更新不生效的问题
  • 实现原理：
    • 打包时根据文件内容生成唯一哈希值作为文件名部分
    • 内容不变则哈希值不变，利用浏览器缓存
    • 内容变化则哈希值变化，强制浏览器重新请求
  • 开发影响：完全由webpack自动处理，开发者无需手动干预
4）资源路径

  • 资源模块与代码、样式模块的区别

    • 模块分类：在webpack中所有内容都被视为模块，其中代码模块（JS）和样式模块（CSS/LESS）是特殊模块，其他如图片、文本等都被归类为资源模块
    • 特殊处理：代码和样式模块会经过特殊编译处理，而资源模块会保持原始格式但路径会被修改

  • 资源模块路径问题

    • 路径差异：资源模块在源代码中的路径（如./src/a.png）和打包后的路径（如./dist/assets/f3ad7.png）完全不同
    • 开发影响：这种差异导致开发时无法预知最终路径，例如源代码写./a.png但打包后实际路径会变成哈希值文件名

  • CSS中背景图片路径处理

    • 自动转换：在CSS中使用的资源路径（如background: url(./a.png)），webpack会智能识别并自动转换为打包后的正确路径
    • 验证方式：通过开发服务器运行可以看到实际生效的路径已被替换为类似/assets/f3ad7.png的形式
    • 无需干预：开发者不需要手动处理CSS中的资源路径转换，webpack会完成这项工作

  • JS动态使用路径问题

    • 识别限制：webpack无法识别JS中动态拼接的字符串路径（如img.src = ‘./a.png’）
    • 问题表现：这种写法打包后路径不会自动转换，导致404错误，因为实际文件已被重命名
    • 错误示例：直接设置$(‘<img>‘).prop(‘src’,’./a.png’)会导致图片无法加载

  • 正确导入资源方式

    • 模块化导入：需要通过ES模块语法导入资源文件（如import url from ‘./a.png’）
    • 路径获取：导入后会得到资源打包后的真实路径，可以赋值给DOM元素的src属性
