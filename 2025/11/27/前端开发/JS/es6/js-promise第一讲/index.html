一、promise基础

1. 例题:表白短信回调地狱问题
1）问题场景描述

*

  • 串行表白策略：邓哥采用串行方式向多位女神表白，依次等待前一位女神的回应后再决定是否向下一位表白
  • 女神名单：李建国、王富贵、周聚财、刘人勇（共4位）
  • 成功概率：每次表白有10%的成功几率（Math.random() ≤ 0.1）
2）回调函数实现

  • 函数结构：
  • 参数说明：
    • name: 女神姓名
    • onFulfilled: 成功回调函数
    • onRejected: 失败回调函数
  • 模拟延迟：使用setTimeout模拟1秒的等待回复时间
3）回调地狱示例

  • 嵌套调用：
  • 问题表现：
    • 每层失败回调中嵌套下一个sendMessage调用
    • 代码向右无限延伸，形成”金字塔”形状
    • 最终出现大量闭合括号，难以维护
4）回调地狱的危害

  • 可读性差：代码呈现深度嵌套结构，逻辑难以追踪
  • 维护困难：添加/删除中间步骤需要重构整个嵌套结构
  • 错误处理复杂：需要在每个层级单独处理错误
  • 扩展性差：女神数量增加时代码复杂度呈指数增长
5）传统异步处理模式
  • 常见异步场景：
    • setTimeout/setInterval：定时器回调
    • addEventListener：事件监听回调
    • AJAX请求：成功/失败回调
  • 共同特点：都采用回调函数处理异步结果
  • 局限性：多个异步操作串联时必然形成回调嵌套
6）Promise的引入

  • 解决目标：避免回调地狱，使异步代码更清晰
  • 规范来源：Promise A+社区规范
  • 核心思想：
    • 将异步操作封装为Promise对象
    • 通过链式调用替代嵌套回调
    • 统一处理成功/失败状态
2. promise规范
1）promise API

  • Promise对象

    *

    • 规范定义: Promise是一套专门处理异步场景的规范，能有效避免回调地狱的产生，使异步代码更加清晰、简洁、统一。

    • 规范起源: 最早诞生于前端社区，规范名称为PromiseA+，得到开发者广泛响应。

    • 核心思想: 英文”promise”在商务英语中表示约定，即”如果…就…”的逻辑关系。

    • 异步任务模型

      • 任务对象化: 所有异步场景都可看作异步任务，在JS中表现为Promise对象（又称任务对象），如发送表白短信、远程登录、延时弹窗等场景。
      • 阶段与状态:
        • 两个阶段: 未决阶段(unsettled)和已决阶段(settled)
        • 三种状态: 挂起状态(pending)、完成状态(fulfilled)、失败状态(rejected)
      • 状态转换规则:
        • 任务总是从未决阶段变为已决阶段，不可逆行
        • 状态总是从pending变为fulfilled或rejected，不可逆行
        • 任务一旦完成或失败，状态就固定不变

    • 状态转换术语

      • resolve: 从pending到fulfilled的状态转换过程
      • reject: 从pending到rejected的状态转换过程
      • 相关数据:
        • 任务完成时可能携带相关数据（如登录成功返回的用户信息）
        • 任务失败时可能携带失败原因（如网络请求的错误信息）

    • 可以针对任务进行后续处理

      • onFulfilled: 针对完成状态的后续处理函数，可获取任务完成的相关数据
      • onRejected: 针对失败状态的后续处理函数，可获取任务的失败原因
      • 处理特点:
        • 后续处理可链式添加（如表白被拒后转向下个目标）
        • 每个处理函数都能获取到对应阶段的状态数据

    • 实现标准: ES6的Promise API完全实现了PromiseA+规范，是现代JavaScript处理异步操作的标准方式

  • Promise API示例

    • Promise对象的基本用法

      • Promise对象的创建与状态
        • 创建方式：通过new Promise()构造函数创建任务对象，创建后立即进入pending状态
        • 执行器函数：构造函数接收一个执行器函数，该函数会立即执行，用于描述任务的具体流程
        • 状态转换：
          • resolve(data)：将任务从pending变为fulfilled状态，data为传递的相关数据
          • reject(reason)：将任务从pending变为rejected状态，reason为失败原因
      • Promise的生命周期
        • 未决阶段(unsettled)：任务创建后的初始pending状态
        • 已决阶段(settled)：任务最终状态，分为：
          • fulfilled：通过resolve(data)进入的完成状态
          • rejected：通过reject(reason)进入的失败状态
      • 任务处理流程示例
        • 任务描述：通过执行器函数详细描述任务流程
          • 示例中模拟了”百米短跑”任务，包含开始、随机持续时间、成功/失败判断
        • 状态转换触发：
          • 成功时调用resolve(duration)传递跑步时间
          • 失败时调用reject(‘脚扭伤了！’)传递失败原因
      • 后续处理then方法
        • 方法结构：pro.then(onFulfilled, onRejected)
          • onFulfilled：任务完成时自动执行的函数，接收data参数
          • onRejected：任务失败时自动执行的函数，接收reason参数
        • 处理示例：
          • 成功时输出”on yeah!我跑了”+跑步时间
          • 失败时输出”不好意思，”+失败原因
      • 邓哥表白函数改造
        • 改造要点：
          • 函数返回Promise对象而非直接处理回调
          • 使用resolve/reject处理成功/失败情况
          • 外部通过then方法处理不同结果
        • 优势：
          • 分离任务执行和结果处理逻辑
          • 支持链式调用，避免回调地狱
          • 统一错误处理机制
      • 关键记忆点
        • 立即执行：Promise构造函数接收的执行器函数会立即执行
        • 状态不可逆：一旦从pending变为fulfilled或rejected，状态不可再改变
        • 参数传递：resolve/reject只能接收一个参数，多个数据需封装为对象
        • 异步处理：then方法的回调函数总是异步执行，即使Promise立即resolve

    • 例题：sendMessage改造

      • Promise基本结构：
        • 通过new Promise((resolve, reject) => {…})创建Promise对象
        • .then()方法接收两个回调函数：成功回调(data)=>{}和失败回调(reason)=>{}
        • 示例中成功输出”on yeah!我跑了”+数据，失败输出”不好意思”+原因
      • 异步任务改造要点：
        • 原函数sendMessage接收三个参数：name, onFulfilled, onRejected
        • 改造后只需接收name参数，返回Promise对象
        • 原回调逻辑改为Promise的resolve/reject调用
        • 成功概率模拟：使用Math.random() <= 0.1实现10%成功率
      • 调用方式变化：
        • 原调用方式：sendMessage(name, successFn, failFn)
        • Promise化后：sendMessage(name).then(successFn, failFn)
        • 可直接链式调用，无需先保存到变量
        • 成功/失败处理逻辑保持不变，仅调用形式变化

    • 例题：delay函数编写

      *

      • 函数要求：
        • 接收一个duration参数（毫秒数）
        • 返回Promise对象，在指定时间后完成
        • 无失败情况，只需调用resolve()
        • 等效于Promise版的setTimeout
      • 实现要点：
        • 核心结构：return new Promise(resolve => {…})
        • 内部使用setTimeout(resolve, duration)实现延迟
        • 不需要reject参数，因为延迟不会失败
        • 调用示例：delay(1000).then(()=>console.log(“finish”))
      • Promise状态机制：
        • 初始状态：pending（挂起）
        • 成功转换：调用resolve(data)变为fulfilled（完成）
        • 失败转换：调用reject(reason)变为rejected（拒绝）
        • 状态转换后不可逆
        • 后续处理：通过.then()注册onFulfilled和onRejected回调

    • 例题：createImage函数编写

      • Promise基础概念
        • 创建Promise：通过new Promise()构造函数创建，接收一个执行器函数(resolve, reject) => {}
        • 异步任务封装：Promise用于封装需要等待的异步操作，如setTimeout、图片加载等

  • 错误表现：当尝试加载受保护的图片时，控制台会显示”Failed to load resource: the server responded with a status 403”的错误信息
1. Promise基础
1）例题：图片加载任务

  • 任务要求：
    • 创建一个返回Promise的函数，处理图片加载
    • 成功时返回图片DOM元素，失败时返回错误原因
    • 使用img元素的load和error事件监听加载状态
  • 实现要点：
    • 创建Promise：使用new Promise((resolve, reject) => {…})结构
    • 事件处理：
      • img.onload：调用resolve(img)返回加载完成的图片元素
      • img.onerror：调用reject(e)返回错误事件对象
    • 灵活处理：函数只负责加载图片，不关心后续处理，由调用者决定成功/失败后的操作
2）例题：图片处理任务

  • 任务1：成功加载后显示图片宽高
    • 使用.then()处理成功情况
    • 通过image.width和image.height获取尺寸
    • 更新p元素的innerHTML显示尺寸信息
  • 任务2：成功加载后将图片添加到容器
    • 使用.then()处理成功情况
    • 通过container.appendChild(image)添加图片到DOM
    • 失败时在控制台输出错误信息
  • 核心思想：
    • Promise让异步任务和处理逻辑分离
    • 同一个加载函数可适配不同处理需求
    • 网络状况影响加载速度，但代码逻辑不变
