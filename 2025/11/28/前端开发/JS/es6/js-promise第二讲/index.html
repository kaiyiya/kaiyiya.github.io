2）catch方法应用
  • catch方法原理
    • 语法糖本质：.catch(onRejected)等同于.then(null, onRejected)
    • 使用场景：当只需要处理失败情况时，使用catch方法代码更简洁
    • 示例对比：
  • 链式调用概念
    • 任务串联：现实中的任务常需要按顺序执行，前一个任务完成才能开始下一个（如：学习→考试→填志愿→入学）
    • 链式特点：
      • 每个.then()都会返回一个新的Promise对象
      • 新Promise状态由后续处理决定：
        • 无相关处理：保持前任务状态
        • 有未执行处理：新任务挂起(pending)
        • 处理执行无错：新任务完成(fulfilled)
        • 处理执行有错：新任务失败(rejected)
        • 返回新Promise：状态与返回的Promise一致
    • 优势：使异步代码具有更强的表达力和可读性
3）Promise链式调用原理

  • 链式调用本质

    • 任务连续性：每个.then()都代表一个新的异步任务，如”学习→考试→出成绩”的任务链
    • 返回机制：.then()方法必定返回新Promise对象，后续处理本身就是一个独立任务
    • 代码示例：
  • 新Promise状态决定机制
    • 无后续处理情况
      • 示例说明：前任务resolve(123)，则新Promise同样为fulfilled状态且携带数据123
    • 有未执行处理
      • 挂起状态：当后续处理函数已注册但尚未执行时，新Promise处于pending状态
      • 执行时机：处理函数会进入微任务队列，等待当前宏任务执行完毕
    • 已执行处理
      • 成功情况：
        • 处理函数无报错 → 新Promise变为fulfilled
        • 数据为处理函数的返回值
      • 失败情况：
        • 处理函数抛出异常 → 新Promise变为rejected
        • 数据为Error对象
      • 返回Promise：
        • 若处理函数返回Promise，新Promise将与该返回对象状态同步
  • 链式调用优势
    • 流程控制：通过.then().catch()实现成功/失败分支处理
    • 代码组织：
  • 表达力增强：解决传统回调地狱问题，形成线性任务流水线
  • 执行顺序原理
    • 微任务机制：.then()回调进入微任务队列，导致console.log同步代码先执行
    • 状态观察：可通过打印Promise对象实时查看状态变化（pending→fulfilled）

  • 例题：Promise链式调用中的失败处理

    • Promise链式调用的基本规则
      • 状态传递机制：
        • 每个.then()方法必定返回一个新的Promise对象
        • 新Promise的状态取决于后续处理：
          • 若无相关处理，新任务状态与前任务一致（包括数据和状态）
          • 若有处理但未执行，新任务保持pending状态
          • 若已执行处理，则根据处理结果确定新状态：
            • 处理过程无错 → 新任务fulfilled，数据为处理返回值
            • 处理过程有错 → 新任务rejected，数据为异常对象
            • 返回另一个Promise → 新任务与该Promise状态一致
    • 失败处理的实际应用
      • 处理逻辑：
        • 未处理失败：错误会继续向下传递，最终导致UnhandledPromiseRejectionWarning
        • 已处理失败：通过.catch()或.then的第二个参数处理后：
          • 处理过程正常完成 → 返回fulfilled状态的Promise
          • 处理过程抛出异常 → 返回rejected状态的Promise
      • 关键原则：
        • 后续任务的状态完全取决于处理过程是否出错
        • 无论前任务是成功还是失败，只要正确处理就不会影响后续链式调用

    • 执行过程：
      • pro1被reject后触发catch处理
      • catch回调正常执行（无报错）→ pro2变为fulfilled
      • 若catch回调中出现异常（如a.toString()），pro2会变为rejected
    • 核心要点：
      • 前任务的失败状态可以被正确处理”消化”
      • 处理函数本身的执行情况决定后续Promise状态
      • 必须始终处理Promise链中的潜在拒绝，避免未捕获的异常

  • 例题：Promise状态应用

    • Promise链式调用状态分析
      • 初始状态判断：
        • 所有新建的Promise初始状态都是pending
        • 链式调用中后续Promise的状态依赖于前一个Promise的状态
        • 示例中pro1、pro2、pro3初始状态均为pending
      • 状态传递机制：
        • 前一个Promise未决时，后续Promise必然保持pending
        • pro2等待pro1决议，pro3等待pro2决议，形成依赖链
        • 这种链式依赖是Promise的核心特性之一
    • Promise状态变更过程
      • 定时器触发后：
        • 1秒后pro1变为fulfilled，携带数据1
        • pro1.then()回调执行，输出1并返回data+1(即2)
        • pro2因此变为fulfilled，携带数据2
      • 二次状态传递：
        • pro2.then()回调执行，输出2但未显式返回值
        • 默认返回undefined，使pro3变为fulfilled(undefined)
        • 整个过程展示了Promise状态如何沿链传递
    • 最终状态输出
      • 两阶段输出：
        • 初始输出：三个Promise均为pending状态
        • 2秒后输出：
          • pro1: fulfilled(1)
          • pro2: fulfilled(2)
          • pro3: fulfilled(undefined)
      • 面试考点：
        • 重点考察Promise状态变化时机和传递机制
        • 需要明确每个then()回调的返回值如何影响后续Promise
        • 定时器的存在增加了状态观察的时间维度
    • 核心知识点总结
      • 状态判定三要素：
        • 新建Promise初始必为pending
        • 前序Promise未决时，后续Promise必为pending
        • then()回调的返回值决定后续Promise的最终状态
      • 易错点提醒：
        • 忽略定时器导致的异步时序问题
        • 未正确处理then()回调的返回值
        • 混淆Promise本身和它携带的数据
      • 调试技巧：
        • 使用console.log分阶段输出Promise状态
        • 通过setTimeout观察不同时间点的状态变化
        • 重点记录每个Promise的[[PromiseState]]和[[PromiseResult]]

  • 例题:Promise失败输出应用

    • Promise状态传递机制
      • 初始状态规则：当Promise链中前一个任务未完成时，后续所有Promise状态均为pending。如示例中promise1未完成时，promise2和promise3都显示pending状态。
      • 状态传递规则：
        • 若Promise未被对应状态处理（如成功状态未设置then回调），则新Promise继承原状态和数据。如promise1完成后promise2未设置then回调，则promise2状态与promise1相同（fulfilled，数据为1）。
        • 若处理函数运行无错，则新Promise状态为fulfilled，数据为函数返回值。如promise3处理函数返回undefined，最终状态即为fulfilled undefined。
      • 错误处理特性：当链中某环节抛出错误（如throw 3），后续代码不会执行，该Promise状态变为rejected，错误原因即为抛出值（3）。
    • 例题：链式调用输出分析
      • 题目解析
        • 初始输出：三个Promise初始状态均为pending，控制台首先输出”pending pending pending”。
        • 错误传递：当pro1.then中抛出错误3时，pro2状态变为rejected（原因3），由于pro3未设置catch处理，继承pro2的rejected状态和原因。
        • 最终输出：1秒后输出pro1状态（fulfilled 1），pro2和pro3状态（rejected 3），控制台显示”1 3 3”。
        • 警告说明：未处理的Promise rejection会触发Node.js警告，但不影响程序执行。

  • 例题:Promise链式调用输出应用

    • 链式调用本质
      • 链式原理：每个.then()或.catch()都返回新Promise，因此可以连续调用。如示例中promise1.then()返回promise2，promise2.catch()返回promise3。
      • 状态决定因素：
        • 若前一个Promise未被对应处理（如成功状态遇到.catch()），新Promise继承原状态。如promise2成功时，promise3因未设置then回调而继承fulfilled状态。
        • 处理函数返回值决定新Promise数据，无return则数据为undefined。
      • 执行顺序：先同步输出初始pending状态，再按异步顺序输出处理结果。如示例最终输出顺序为”pending→1→2”。
    • 深度链式调用分析
      • 题目解析
        • 状态传递链：
          • promise1立即fulfilled（数据1）
          • promise2处理函数返回2，状态变为fulfilled 2
          • promise3因未处理promise2成功状态，继承fulfilled 2
          • promise4处理函数无返回值，状态为fulfilled undefined
        • 输出逻辑：
          • 第一个.then()输出promise1结果1
          • 最后一个.then()输出promise3结果2
        • 最终状态验证：通过setTimeout可获取最终Promise状态为fulfilled undefined。

  • Promise链式调用原理

    • 链式调用本质：每个.then()或.catch()都会返回一个新的Promise对象，形成任务处理链
    • 状态传递规则：
      • 无处理函数：新Promise状态与前任务一致（如只有.then()但前Promise是rejected）
      • 有处理函数：根据处理函数执行结果决定新状态
        • 正常完成：新Promise状态为fulfilled，数据为返回值
        • 抛出异常：新Promise状态为rejected，数据为异常对象
        • 返回Promise：新Promise状态与该Promise一致
    • 错误处理机制
      • 错误捕获流程：
        • 当.then()中代码报错（如调用undefined.toString()）
        • 会跳过后续.then()，直接寻找最近的.catch()
        • catch处理成功后，后续.then()会继续执行
      • 特殊情形：
        • 若catch中又抛出错误，会继续向下传递
        • 返回普通值（如return 3）会使链式调用恢复为fulfilled状态
    • 状态变化分析
      • 典型分析步骤：
        • 确定初始Promise状态（pro1）
        • 检查.then()回调是否执行（取决于前状态）
        • 判断回调执行是否报错
        • 根据报错与否确定新Promise状态
      • 记忆要点：
        • 报错会使整个Promise链”短路”，直到被catch捕获
        • setTimeout中的console.log(pro)输出的是最终状态

  • 链式调用实战案例

    • 例题：错误抛出与捕获
      • 题目解析
        • 初始Promise直接throw new Error(1) → pro1 rejected
        • .then()未处理rejected状态 → pro2保持rejected
        • catch()重新抛出错误 → pro3保持rejected
        • 最后一个.then()不执行（因前序为rejected且无catch）
        • 最终无输出，整个链以rejected状态结束
    • 例题：返回Error对象
      • 题目解析
        • 关键区别：return new Error() vs throw new Error()
        • return Error对象不会触发rejected状态
        • 执行流程：
          • pro1 fulfilled(1)
          • .then()正常返回Error对象 → pro2 fulfilled(Error)
          • 下一个.then()正常执行 → 输出Error对象
          • 最终pro4 fulfilled(undefined)

  • Promise核心原理总结

    • 三大核心规则：
      • 每次调用.then/.catch都创建新Promise
      • 新Promise状态由回调执行结果决定：
        • 正常返回 → fulfilled
        • 抛出异常 → rejected
        • 返回Promise → 继承其状态
      • 错误会沿链向下传递直到被捕获
    • 编程实践：
      • 推荐链式写法：pro.then().catch()
      • 需要串行处理时：pro.then().then()…catch()
      • 避免在then中混用业务逻辑和错误处理
2. 平时开发写代码
1）常见的任务处理代码写法
  • 基础处理模式：
    • 任务成功后执行处理1，失败则执行处理2：pro.then(处理1).catch(处理2)
    • 任务成功后依次执行处理1、处理2：pro.then(处理1).then(处理2)
    • 任务成功后依次执行处理1、处理2，若任何步骤出错则执行处理3：pro.then(处理1).then(处理2).catch(处理3)
  • 开发实践：
    • 实际开发中不需要过度思考Promise内部状态变化，只需按业务逻辑编写成功/失败处理
    • 链式调用时前一步处理结果会自动传递给下一步处理函数
    • 错误处理可以集中放在链式调用的最后通过catch捕获
2）then方法与catch方法的使用
  • then方法特性：
    • 必定返回新的Promise对象
    • 新任务状态取决于后续处理：
      • 无相关处理时保持原任务状态
      • 有未执行处理时状态为挂起
      • 处理无错时状态为完成，数据为返回值
      • 处理有错时状态为失败，数据为异常对象
      • 返回任务对象时状态与该对象一致
  • catch方法：
    • 专门处理Promise链中的错误
    • 可以捕获链式调用中任意位置的错误
    • 实际开发中常用于统一错误处理
3）链式调用的理解与应用
  • 链式调用本质：
    • 每个then/catch都返回新的Promise对象
    • 前一步处理结果自动成为下一步的输入
    • 类似”下一步、下一步”的连续处理流程
  • 开发技巧：
    • 对于连续异步操作，使用链式调用可避免回调地狱
    • 错误处理可以集中放在调用链末尾
    • 实际开发中更关注业务逻辑而非Promise内部状态

      2. 打印拒绝原因后继续向王富贵发送消息
      3. 若再次失败，继续向周聚财发送
      4. 最后向刘人勇发送，任一环节成功即终止
  • 状态传递：每个.catch()返回的新Promise状态与内部sendMessage返回的Promise状态一致
  • 概率设置：通过Math.random() <= 0.1设置10%的成功率模拟现实场景
3）运行结果分析
  • 成功情况：当任一环节成功时，直接执行最后的.then()成功回调
  • 全部失败：当所有环节都失败时，执行最后的失败回调
  • 调试技巧：可通过调整成功率（如从0.1改为0.3）来测试不同场景
4. 例题:catch方法应用
1）题目分析
  • 题目解析
    • 初始状态：promise1创建时为pending状态，1秒后被拒绝
    • 链式处理：promise2通过.catch()处理promise1的拒绝，返回值为2
    • 输出时机：
      • 立即输出时两个Promise都是pending
      • 1秒后promise1变为rejected
      • promise2变为fulfilled，值为2
    • 答案
      • 初始输出：两个Promise都是pending状态
      • 最终输出：promise1为rejected(undefined)，promise2为fulfilled(2)
2）核心知识点

  • 状态转换规则：
    • Promise初始状态为pending
    • 通过resolve()转为fulfilled
    • 通过reject()转为rejected
  • 链式调用特点：
    • .catch(fn)等价于.then(null, fn)
    • 前一个Promise的状态决定后续处理是否执行
    • 处理函数的返回值决定新Promise的状态和数据

二、Promise对象与链式调用

1. Promise的基本状态

  • 未决阶段(unsettled): Promise对象创建后的初始状态，此时操作尚未完成，处于挂起状态(pending)
  • 已决阶段(settled): 操作完成后进入的最终状态，分为两种：
    • 完成状态(fulfilled): 通过resolve(data)触发，执行onFulfilled回调
    • 失败状态(rejected): 通过reject(reason)触发，执行onRejected回调
2. catch方法原理

  • 语法糖本质: .catch(onRejected) 完全等效于 .then(null, onRejected)
  • 错误处理机制: 专门用于捕获Promise链中的错误，避免在每个then中都写错误处理
  • 链式特性: 可以继续在catch后接then或另一个catch，形成连续的错误处理流程
3. 链式调用核心规则

  • 必然返回新Promise: 每次调用.then()或.catch()都会生成新的Promise对象
  • 状态传递机制:
    • 无后续处理时: 新Promise保持前任务状态和数据
    • 有未执行处理时: 新Promise处于pending挂起状态
    • 处理执行后:
      • 无错：新Promise为fulfilled，数据为处理函数返回值
      • 有错：新Promise为rejected，数据为异常对象
      • 返回Promise：新任务与该Promise状态数据一致
4. 邓哥解决方案案例

  • 链式错误处理模式:
  • 关键特点:
    • 每个catch处理前序失败并尝试新方案
    • 最终then处理成功或完全失败的情况
    • 展示了Promise链强大的异步流程表达能力
  • 学习建议: 遇到理解困难时，需要投入足够时间反复实践，不要期望快速掌握，Promise机制需要通过实际编码逐步内化。
