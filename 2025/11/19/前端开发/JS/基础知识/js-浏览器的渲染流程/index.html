js-浏览器的渲染流程

飞向月球的猫2025-11-192025-11-26

一、浏览器的渲染流程

1. 解析HTML
1）解析HTML的步骤

  • 基本流程：浏览器从网络或本地获取HTML源代码后，会从上到下逐行解析代码
  • 解析机制：将HTML字符串转换为可理解的DOM元素结构，类似编译原理中的词法分析过程
  • DOM树生成：边解析边构建DOM树，JS中获取的DOM元素即来自这棵树
2）HTML文档的获取途径

  • 网络获取：通过URL从服务器获取（如访问www.jd.com），实际所有请求都经过网络层，包括本地开发时的127.0.0.1
  • 本地获取：直接双击HTML文件用浏览器打开（file://协议），不经过网络请求
  • 开发工具差异：Live Server等工具仍通过本地服务器提供网络访问
3）解析过程中的阻塞

  • 阻塞资源：遇到CSS/JS会立即停止HTML解析，优先处理这些资源
  • 非阻塞资源：图片、音视频等不会阻塞解析，采用异步加载方式
  • 等待机制：CSS文件下载慢时会真实等待（如5秒），期间不继续解析后续HTML
4）CSS与JS的加载顺序

  • CSS前置原因：
    • 避免用户看到无样式的”丑陋页面”
    • 防止页面渲染后样式突变造成的闪烁现象
    • 通过3G网络模拟可见明显闪烁效果（默认样式→最终样式）
  • JS后置原因：
    • 优先保证页面内容快速呈现
    • 交互功能可以稍后加载（用户浏览需要反应时间）
    • 避免阻塞关键渲染路径
5）页面闪烁的避免

  • 重现方法：将CSS链接放在body末尾，通过网络限速观察
  • 根本原因：浏览器逐步渲染导致的无样式内容闪现（FOUC）
  • 解决方案：
    • 所有CSS放在 <head>内
    • 关键CSS内联写入HTML
    • 使用loading占位符过渡
  • 性能权衡：虽然CSS前置会轻微增加首屏时间，但用户体验更连贯
2. 生成DOM树
1）浏览器渲染流程解析

  • 关键步骤：解析HTML → 生成DOM树 → 生成渲染树 → 布局(layout/reflow) → 重绘(repaint)
  • 优化技巧：将JS代码写在HTML文档最后，让浏览器先呈现页面再执行JS交互功能
  • 并行处理：浏览器会一边解析HTML一边生成DOM树，不是等全部解析完才开始构建
2）DOM树生成过程

  • 构建方式：浏览器采用增量构建方式，解析到哪个标签就生成对应节点
  • 节点关系：最顶部是document对象，通过children属性可获取html元素，再通过children获取head/body
  • JS访问时机：若JS写在文档开头，此时DOM树未构建完成，无法获取后续元素（如h1）
  • 验证示例：演示在head中通过document.querySelector(‘h1’)获取不到元素的实际情况
3）DOMContentLoaded事件

  • 触发条件：当DOM树完全生成后触发，不等待外部资源（如图片）加载
  • 注册方式：必须使用document.addEventListener()，不支持onDOMContentLoaded写法
  • 典型应用：将JS代码写在文档头部时，需把DOM操作放入此事件回调中

4）load事件与资源加载
  • 触发时机：需等待页面所有外部资源（图片、CSS等）加载完成
  • 注册方式：可通过window.onload或window.addEventListener(‘load’)注册
  • 与DOMContentLoaded区别：
    • DOMContentLoaded只关心DOM结构
    • load关心所有资源加载状态
  • 单资源监听：可为单个资源（如图片）注册onload事件
  • 网络调试技巧：通过开发者工具模拟慢速网络，观察两个事件的触发顺序差异
3. 生成渲染树

  • 并行处理机制：浏览器一边生成DOM树，一边计算DOM树中每个节点的样式规则，最终形成渲染树
  • CSS计算时机：CSS属性的计算过程发生在生成渲染树阶段，包括：
    • 先看声明值
    • 处理层叠冲突
    • 使用继承规则
    • 应用默认值
  • 完整属性计算：每个DOM节点都会计算出一套完整的CSS属性集合，包含所有可能的样式属性
4. 布局layout/重排reflow
1）布局layout/重排reflow定义

  • 核心功能：计算渲染树中每个元素最终的尺寸和位置，包含：
    • 元素在视口中的显示位置
    • 元素的实际显示尺寸（考虑遮挡、滚动条位置等）
    • 窗口大小和滚动位置关系
  • 反复执行特性：该过程会在页面运行期间不断重复，例如：
    • 滚动条滚动时
    • 图片加载完成后
    • 元素位置/尺寸变化时
  • 专业术语：该步骤在不同浏览器中被称为layout或reflow（回流/重排）
2）重排reflow的触发操作

  • 直接触发操作：
    • 获取元素的尺寸和位置（如clientHeight、offsetTop等）
    • 直接改变元素的尺寸和位置（如修改width、left等样式）
  • 间接触发操作：
    • 修改图片src导致图片尺寸变化
    • 改变字体大小影响元素布局
    • 任何会导致元素尺寸/位置变化的DOM操作
  • 性能影响：重排是非常耗时的操作，会显著影响页面性能
3）浏览器性能优化策略

  • 延迟重排机制：对于连续导致重排的JS代码，浏览器会将重排延迟到最后统一执行
    • 示例：连续修改width、height、left、top只会触发一次重排
  • 强制立即重排：如果在延迟期间遇到获取尺寸/位置的代码，浏览器会强制立即重排
    • 原因：需要确保获取的值是最新的准确值
  • 优化效果：将多次重排合并为一次，显著提升性能

  • 性能影响：上述代码会导致两次重排（读取clientHeight时一次，最后统一处理时一次）
  • 根本原因：浏览器必须确保读取的尺寸/位置值是准确的最新值
5）开发中的性能优化建议

  • 减少读取操作：避免频繁读取元素尺寸/位置，特别是在事件处理函数中
  • 缓存测量值：对于不变的尺寸/位置，应该提前缓存到变量中复用
  • 批量修改：将多个样式修改集中在一起执行，利用浏览器的延迟重排机制
  • 实践案例：在mousemove事件处理中，避免每次事件都读取元素尺寸
  • 开发意识：需要了解重排机制，在代码中主动考虑性能优化
5. 重绘repaint
1）重绘的定义与过程
  • 绘制机制：浏览器在reflow(重排)过程中同步生成对应图形并绘制到页面，这个绘制过程称为repaint(重绘)
  • GPU加速：绘制过程完全由GPU完成，速度非常快
  • 视觉原理：类似于手速极快的老师不断擦除重画，人眼会将其感知为连续动画效果
2）重绘与重排的关系

  • 必然联系：所有导致reflow(重排)的代码必定会触发repaint(重绘)，因为元素尺寸/位置变化后必须重新绘制
  • 效率对比：仅导致repaint的代码效率远高于导致reflow的代码，因为避免了布局计算的开销
  • 性能关键：重排需要重新计算元素几何属性，而重绘只需更新像素绘制
3）仅导致重绘的操作
  • 外观修改：仅改变元素视觉表现而不影响布局的CSS属性：
    • 背景颜色改变
    • 字体颜色改变
    • 圆角边框设置(盒子实际尺寸不变)
    • 背景图更换
  • 核心特征：这些修改不会触发元素尺寸/位置的重新计算，但需要更新视觉呈现
4）重绘在动画中的应用

  • 传统JS动画：通过定时器逐步修改元素位置(如left/top)，每次修改都会触发reflow+repaint
  • CSS3动画优势：
    • 通过transform等属性实现视觉位移，不改变实际布局
    • 仅触发repaint，避免reflow带来的性能损耗
    • GPU加速使动画更加流畅
  • 开发建议：优先使用CSS3实现动画效果，特别是在移动端等性能敏感场景

二、浏览器解析HTML流程

1. 解析机制
  • 基本流程: 浏览器从网络或本地文件中获取HTML源代码后，会从上到下逐行解析代码
  • 阻塞现象: 当解析过程中遇到CSS或JS资源时，会暂停HTML解析（产生阻塞），优先处理CSS解析或JS执行
  • 文档声明: 有效的HTML文档应以 <!DOCTYPE html>声明开头，这是浏览器识别文档类型的关键标记
2. 典型HTML结构
  • 语言声明: <html lang="en">指定文档使用的自然语言
  • 元数据区域: <head>标签内包含的 <meta>标签定义字符集(charset=”UTF-8”)、兼容性设置(X-UA-Compatible)和视口配置(viewport)
  • 资源加载: 通过 <link rel="stylesheet">加载CSS，通过 <script src="">加载JS，这些外部资源会触发解析阻塞
3. 学习建议
  • 理论记忆: 对于理论性知识需要通过笔记或口头复述强化记忆

三、知识小结

知识点 核心内容 考试重点/易混淆点 难度系数
浏览器渲染流程 解析HTML → 生成DOM树 → 生成渲染树 → 布局/重排 → 重绘 解析阻塞机制（CSS/JS阻塞 vs 图片异步加载） ⭐⭐⭐⭐
关键渲染路径优化 CSS放头部（避免FOUC）、JS放底部（减少阻塞） DOMContentLoaded vs load事件触发时机差异 ⭐⭐⭐
重排(reflow)机制 计算元素几何属性（尺寸/位置），触发条件： 1. 获取/修改尺寸属性; 2. 修改DOM结构; 3. 窗口缩放/滚动 浏览器优化策略（异步队列批量处理）;强制同步布局问题（连续读写属性） ⭐⭐⭐⭐⭐
重绘(repaint)机制 像素级绘制（不涉及几何计算），触发条件： - 颜色/背景等视觉样式变更; - CSS3变形/过渡（GPU加速） 与重排的性能差异（重绘开销更低） ⭐⭐
性能优化实践 1. 避免频繁读写DOM; 2. 使用CSS3动画替代JS动画; 3. 批量DOM操作（文档片段） 防抖/节流在事件处理中的应用场景 ⭐⭐⭐⭐
浏览器事件模型 DOMContentLoaded（DOM就绪）; load（资源加载完成）; 单个资源的onload事件 事件注册方式差异（addEventListener更可靠） ⭐⭐⭐
