一、事件传播机制

1. 事件传播机制
1）事件传播的基本原理
  • DOM层级关系: 页面元素是层层嵌套的结构，从最外层的window开始，依次是document → html → body → container → button
  • 事件触发本质: 点击某个元素时，实际上同时触发了其所有祖先元素的事件（例如点击button等同于点击了container/body/html/document/window）
2）事件传播三阶段
  • 捕获阶段：事件从window开始向下传递，经过document → html → body → div，直到目标元素（button）
  • 目标阶段：事件到达实际被操作的元素（如被点击的button）
  • 冒泡阶段：事件从目标元素向上冒泡，依次经过div → body → html → document → window

    • 日常开发中较少主动使用，但特定场景下（如事件委托）能发挥关键作用
    • 理解该机制可以避免事件处理中的意外行为
2. 事件传播机制
1）监听DOM事件

  • 事件传播机制

    • 三个阶段：事件捕获阶段 → 目标阶段 → 事件冒泡阶段
    • 默认监听阶段：
      • 使用onclick或addEventListener不传第三个参数时，默认监听冒泡阶段
      • 示例：document.body.onclick = function(){} 监听的是body的冒泡阶段
    • 捕获阶段监听：
      • 通过addEventListener的第三个参数设为true实现
      • 示例：div.addEventListener(‘click’, function(){}, true)会在捕获阶段触发
    • 执行顺序：
      • 点击按钮时：按钮(目标阶段) → 父元素div → body（冒泡阶段）
      • 若在捕获阶段监听：body → 按钮 → div（捕获阶段优先执行）

  • 事件对象属性

    • target与this区别：
      • e.target：始终指向触发事件的原始元素（目标阶段元素）
      • this：指向当前正在处理事件的元素（注册事件的元素）
      • 示例：给body注册事件时点击按钮，this是body，e.target是button
    • 阻止事件传播：
      • e.stopPropagation()：阻止事件继续向上冒泡
      • 应用场景：在按钮事件中调用后，父元素将不会收到该事件
    • 事件委托原理：
      • 利用冒泡机制，只需在父元素注册一个事件处理函数
      • 通过e.target判断实际触发事件的子元素

  • 事件委托示例

    • 传统方式缺点：
      • 需要为每个按钮单独绑定事件，产生大量函数对象
      • 动态新增元素时需要重新绑定事件
    • 委托实现步骤：
      • 在父容器（如div.container）注册点击事件
      • 通过e.target.tagName判断点击的是否为BUTTON元素
      • 使用e.target.innerText获取按钮文本
    • 优势体现：
      • 无论页面有多少按钮，只需一个事件处理函数
      • 动态添加按钮无需额外绑定，自动继承事件处理
2）例题：事件委托方式代码

  • 事件委托的基本概念

    • 核心思想：将事件监听器绑定在父元素上，利用事件冒泡机制处理子元素的事件。
    • 适用场景：
      • 子元素数量多时避免性能浪费
      • 动态生成的子元素需要事件处理
    • 优势：只需注册一次事件监听，代码更简洁高效。

  • 事件传播机制详解

    • 事件类型区别：
      • mouseenter/mouseleave：不冒泡
      • mouseover/mouseout：会冒泡
    • 实际应用：
      • 需要事件委托时选择冒泡类型事件
      • 需要精确控制时选择不冒泡事件

  • 事件委托的两种典型应用场景

    • 性能优化：
      • 当子元素数量特别多时，避免为每个子元素单独绑定事件
    • 动态元素处理：
      • 对动态创建的元素，无需在创建时单独绑定事件
    • 判断技巧：
      • 通过e.target的特征判断实际交互元素
      • 常用判断条件包括tagName、className等
3）例题:鼠标移入事件不冒泡

  • mouseenter事件特性

    • 不冒泡特性: mouseenter事件不会向上冒泡，当鼠标移入元素时，只会触发当前元素的事件处理函数，不会触发父元素的同类事件。
    • 示例现象: 当鼠标移入inner元素时，控制台只输出”inner mouseenter!”，不会输出”container mouseenter!”，说明事件没有冒泡到父容器。
    • 区域概念: mouseenter和mouseleave事件关注的是”进入/离开整个区域”的行为，而不是具体的冒泡过程。

  • mouseover与mouseenter的区别

    • 冒泡行为差异:
      • mouseover/mouseout: 会冒泡，当鼠标移入子元素时会触发父元素的同类事件，容易造成事件反复触发。
      • mouseenter/mouseleave: 不会冒泡，只在进入/离开元素边界时触发一次。
    • 实际应用建议:
      • 大多数情况下推荐使用mouseenter/mouseleave，避免不必要的重复触发。
      • 需要事件冒泡的特殊场景才使用mouseover/mouseout。

  • 事件传播机制总结

    • 三阶段模型:
      • 捕获阶段：从document向下传播到目标元素
      • 目标阶段：到达事件触发的元素
      • 冒泡阶段：从目标元素向上传播回document
    • 特殊事件类型:
      • 不是所有事件都会冒泡，如mouseenter/mouseleave、focus/blur等事件就不参与冒泡过程。
      • 可以通过event.stopPropagation()方法手动阻止事件继续传播。
